## Query-plan evidence

### Exact Key equality + time range

Representative count query:

```sql
SELECT COUNT(*)
FROM request_usage
WHERE created_at_unix >= ?
  AND api_key_hash = ?;
```

Measured plan from the million-row fixture:

```text
SEARCH request_usage USING COVERING INDEX idx_request_usage_key_created_at_unix (api_key_hash=? AND created_at_unix>?)
```

Focused unit tests (`TestExactKeyQueryPlansUseCompositeIndex`) also require this composite index for representative exact-Key SQL shapes:

- requests recent/limit
- activity sample scope
- percentile sample scope
- models core scope
- timeseries core scope
- issues core scope
- cost-bucket core scope

### Unknown Key (`NULLIF` / `TRIM` semantics)

Unknown scope continues to use:

```sql
NULLIF(TRIM(api_key_hash), '') IS NULL
```

That expression **cannot** perform equality seeks on `idx_request_usage_key_created_at_unix`.

On the small unit-test fixture, SQLite chose:

```text
SCAN request_usage
```

This is accepted planner behavior for small tables. The important invariant is:

- exact Key continues to use `api_key_hash = ?`
- unknown does **not** invent composite equality seeks
- semantics remain empty/whitespace/NULL → `unknown`

No covering-index expansion was added; exact-Key plans already selected the composite index where the SQL shape permits.

## Migration / rollback notes

- Fresh DBs receive the index on first `Open`.
- Existing populated DBs receive the index additively on next `Open`; rows and older indexes remain.
- Repeated `Open` is idempotent (`CREATE INDEX IF NOT EXISTS`).
- An extra unknown index is tolerated and left in place (supports old-binary ignore/rollback reasoning for additive indexes). This slice did **not** execute an older binary against the upgraded file.
- First upgrade on a large production DB will spend time creating the index; operators should keep the existing DB+salt backup discipline.

## Honest limitations

1. Results are one machine, one run, `-benchtime=1x`. Treat them as evidence, not an SLO.
2. Fixture keys/models are synthetic and evenly distributed; production skew can change plan cost.
3. Keys list remains a full-window aggregate and is the slowest report measured here.
4. Unknown-key filtering still depends on `NULLIF(TRIM(...))`; small tables may SCAN.
5. No covering-column expansion was justified beyond the composite index already selected by EXPLAIN.
6. No proxy concurrent-report throughput claim is made in this document.
7. Index creation time on a real production snapshot was not measured in this worktree; the million-row recreate path is the local stand-in.
