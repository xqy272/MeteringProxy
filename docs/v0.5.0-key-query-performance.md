# v0.5.0 Exact-Key Query Performance Evidence

Status: verified on the GPT-reviewed v0.5.0 implementation
Date: 2026-07-22
Scope: additive exact-Key index, production Issues query closeout, query-plan checks, cancellation/WAL tests, and an opt-in 1,000,000-row benchmark.

## Change summary

- Added the durable additive index:

```sql
CREATE INDEX IF NOT EXISTS idx_request_usage_key_created_at_unix
ON request_usage(api_key_hash, created_at_unix);
```

- Kept migration in the existing idempotent `createIndexes` / `Open` flow.
- Replaced the request-usage Issues aggregate/window self-join with one ranked
  window pipeline. Output grouping, severity ordering, count, model-source, and
  latest-sample semantics remain covered by the existing tests.
- Did not change report JSON, the proxy hot path, connection-pool size, or add a
  report cache or dependency.

## Review corrections

The original implementation-slice fixture was not accepted as final evidence:

1. Its timestamp expression covered only about 11.6 days while claiming 30 days.
2. Key and model used the same modulo, so each exact Key saw only one model.
3. Image selection was correlated with row order and did not exercise the
   benchmark Key.

The reviewed fixture now spans an inclusive 30 days, every one of 100 Keys uses
all 100 models, and every Key owns five image rows. The corrected fixture exposed
a real non-linear Issues query: exact-Key 30d took about 3.44 s before the SQL
rewrite and 7.69 ms in the final measured run. All pre-correction benchmark
figures are discarded.

## Query-plan evidence

Representative exact-Key equality plus time range:

```sql
SELECT COUNT(*)
FROM request_usage
WHERE created_at_unix >= ?
  AND api_key_hash = ?;
```

Plan from the final million-row fixture:

```text
SEARCH request_usage USING COVERING INDEX idx_request_usage_key_created_at_unix (api_key_hash=? AND created_at_unix>?)
```

`TestExactKeyQueryPlansUseCompositeIndex` also requires the composite index for
the scope shapes used by requests, activity samples, percentiles, models,
timeseries, Issues, and cost buckets. The benchmark invokes the production
report methods rather than timing stand-in SQL.

Unknown-Key scope remains:

```sql
NULLIF(TRIM(api_key_hash), '') IS NULL
```

That expression cannot perform an equality seek on the composite index. The
test accepts a time-index range or table scan while proving that exact-Key scope
continues to use `api_key_hash = ?`; no prefix or guessed mapping is introduced.

## Cancellation and WAL concurrency evidence

Automated tests cover three distinct boundaries:

1. `TestExactKeyReportsHonorCanceledContextPromptly`
   - pre-canceled Models, Timeseries, Activity, Issues, Requests, CostBuckets,
     and Keys calls return `context.Canceled` promptly.
2. `TestCanceledSQLiteQueryReleasesReadConnection`
   - starts a demonstrably in-flight recursive SQLite query, cancels it, requires
     `context.Canceled`, then proves the single read connection is reusable.
3. `TestExactKeyReportReadDoesNotBlockWriterUnderWAL`
   - holds a read-only transaction while an insert completes, then runs a
     production report concurrently with another insert.

These prove DB-layer cancellation and WAL reader/writer separation. They do not
claim proxy-level throughput or multi-reader scale-out.

## Migration and rollback notes

- Fresh DBs receive the index on first `Open`.
- Existing populated DBs receive it additively on next `Open`; rows and older
  indexes remain.
- Repeated `Open` is idempotent (`CREATE INDEX IF NOT EXISTS`).
- Extra unknown indexes are tolerated and preserved, supporting additive-index
  rollback behavior.
- This slice does not substitute for the separate v0.4.2 binary/database
  upgrade-interruption-rollback rehearsal required before release.
- Index creation on a large DB can extend first startup; the operator must keep
  the DB, WAL/SHM, salt, config, and pricing backup discipline.

## Honest limitations

1. This is one machine, one final `-benchtime=1x` run, not an SLO.
2. The fixture is deliberately even; production Key/model/time skew can change
   planner cost and latency.
3. The global 30d Keys list remains slow at 14.09 s and should be treated as a
   known scaling boundary, not evidence of universally fast reporting.
4. Unknown-Key filtering still uses `NULLIF(TRIM(...))` and may scan.
5. No covering-column expansion, read-pool growth, or cache was justified here.
6. No proxy concurrent-report throughput claim is made by this DB benchmark.
7. Real v0.4.2 snapshot index-creation time remains part of release rehearsal.
