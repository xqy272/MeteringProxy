# MeteringProxy v0.5.0

Client API Key reporting becomes a first-class observability dimension, with a unified read-side reporting core and strict cost/pricing semantics. This is a complete release from the `v0.4.2` baseline; do not deploy intermediate phase builds.

## Highlights

- Client API Key is a first-class report dimension: Keys list, Key Detail, exact-hash scope for Models / Timeseries / Activity / Issues / Requests, optional `key_labels`, and stdin-only `hash-key`.
- Read-side reporting is centralized in `internal/report`; write-side `internal/event` and `internal/store` stay on the metering hot path only.
- One homogeneous request-level cost path serves Overview, Summary, Models, Timeseries, Images, and Keys. `cost_state` / `partial_reasons` make incomplete costs explicit.
- Generic request-level `long_context` tiers, image per-count / size pricing, DeepSeek aliases, and strict pricing validation (unknown fields, negative prices, invalid tiers, alias conflicts).
- Production closeout: `validate`, `/healthz`, `/readyz`, Docker `/healthz` healthcheck with `METERING_PROXY_HEALTH_URL`, low-cardinality report Prometheus metrics, additive Key index, and full upgrade/rollback docs.

## Pricing notes (frozen for this release)

- **Grok 4.5** uses confirmed flat pricing only: input `2.00`, cached input `0.30`, output `6.00` per 1M tokens. No long-context tier is configured or guessed.
- **Gemini 3.1 Pro Preview** appears in default `pricing.yaml` only as a **pricing / long-context tier test example** (threshold 200,000 input tokens). This pricing entry does not select a MeteringProxy runtime or development model.
- Report costs always re-estimate historical observed usage with the **currently loaded** `pricing.yaml`. They are operational estimates, not frozen invoices.
- Default `pricing.yaml` is the copyable release input and is also published as a Release attachment.

## Operator workflow

1. Allocate distinct client API keys for friends, devices, agents, or purposes at CPA / the outer gateway.
2. Compute HMAC hash with `hash-key` from stdin (never `--key`); map hashes to labels via `key_labels` if desired.
3. Inspect Keys → Key Detail → scoped Issues / Requests. Prefix hashes are rejected; use the full 64-char lowercase hex or `unknown`.
4. Treat `partial` / `unavailable` as incomplete trust, not as `$0.00`.

## CLI and health

| Surface | Semantics |
|---|---|
| `serve --config ...` | Production serve path. Bare `--config` remains compatible. |
| `validate --config ...` | Strict offline check of config, pricing, salt, and enabled management key files. No network, no DB open/migrate/write. Success prints only `ok`. |
| `hash-key --config ...` | Reads one plaintext key line from stdin; writes only the 64-char hash to stdout. |
| `GET /healthz` | Process/HTTP liveness only. No DB, pricing, upstream, or CPA probes. |
| `GET /readyz` | Local readiness: loaded config/salt/pricing, SQLite read/write handles, async writer. Does not probe upstream or CPA. |
| Docker healthcheck | Default `METERING_PROXY_HEALTH_URL=http://127.0.0.1:8320/healthz`. Override when listen port is not 8320. |

## Upgrade and rollback

- Back up SQLite + WAL/SHM, salt, config, and pricing together before upgrade.
- Run `validate` with the new image against the operator config before swapping the running container.
- First start of v0.5.0 on a `v0.4.2` DB creates the additive `request_usage(api_key_hash, created_at_unix)` index; reserve a maintenance window for large DBs.
- Rollback is a **full version bundle**: restore the old image **and** the matching old `pricing.yaml` / config. `v0.4.2` does not understand v0.5.0 `long_context` or per-image pricing structures. The new index is safely ignored by the old binary. Salt must stay paired with the database.

## Compatibility and known limits

- Existing API response fields are preserved; Key and cost fields are additive. Invalid filter queries now return stable JSON 400s instead of silent defaults.
- Demo seed (`--seed-demo`) uses synthetic 64-hex hashes and in-memory labels only; never real keys. Labels are not written to SQLite.
- Across two complete runs of the documented million-row fixture, the slowest exact-Key 30d report was about 70 ms; the **global Keys list at 30d remained about 14.1–14.5 s** and is an explicit residual bottleneck, not a target interactive latency.
- Queue overflow still drops metering events rather than blocking LLM traffic. Proxy transparency and privacy invariants are unchanged.

## Not in this release

Key budgets/webhooks, key issuance/revocation UI, online pricing reload, high-cardinality Prometheus labels (key/model/endpoint), SSE allocation overhaul, WebSocket/Realtime upgrade tunnels, full doctor/upstream probe CLI, and destructive migration frameworks remain out of scope for v0.5.0.
