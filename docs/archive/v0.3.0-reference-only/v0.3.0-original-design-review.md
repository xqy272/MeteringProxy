# MeteringProxy v0.3.0 原始设计案审查报告

日期：2026-05-05
审查范围：`docs/v0.3.0-observability-design.md` 完整文档
审查依据：CLIProxyAPI 代码库（`router-for-me/CLIProxyAPI`，main 分支）、MeteringProxy 当前代码（v0.2.0）

通过对 CLIProxyAPI 代码库关键文件的深入审查来验证设计案的可行性与准确性：

- `internal/api/handlers/management/quota.go` — quota 管理端点
- `internal/api/handlers/management/auth_files.go` — auth-files 管理端点

同时审查了 MeteringProxy 当前代码：

## 2. 总体评价

设计案在架构分层、安全边界和防退化约束上考虑成熟。四层观测模型（HTTP 事实 → 协议解析 → Side-channel → 配额快照）自洽且每层有清晰的来源语义和覆盖规则。Section 3 的信息模型、Section 6.4 的 flush 顺序、Section 7.2 的用量合并规则均体现了对转发透明性 invariants 的深刻理解。

但通过 CLIProxyAPI 代码审查发现了若干实现层面的关键问题，其中最严重的是 **request-ID 传播链路在设计案假设与 CLIProxyAPI 实际行为之间存在断裂**。

## 3. 关键发现：Request-ID 传播链路断裂（阻塞级）

### 3.1 设计案的假设

Section 4（请求相关性）和 Section 8（CLIProxyAPI Usage Queue 联动）的核心前提是：MeteringProxy 注入或读取的 `X-Request-ID` header 会出现在 CLIProxyAPI usage queue 的 `request_id` 字段中，从而实现精确匹配。

Section 4.2 明确写道：

> 不使用 `X-Metering-Request-ID` 作为首选 header，因为 CLIProxyAPI usage queue 当前会输出其内部 request id；`X-Request-ID` 更容易被 CLIProxyAPI 请求日志/usage 上下文采纳。

### 3.2 CLIProxyAPI 的实际行为

[`internal/logging/requestid.go`]：

```go
func GenerateRequestID() string {
    b := make([]byte, 4)
    rand.Read(b)
    return hex.EncodeToString(b) // 8 字符随机 hex
}
```

[`internal/logging/gin_logger.go`] 中间件在所有 AI API 路径上调用：

```go
if isAIAPIPath(path) {
    requestID = GenerateRequestID()  // 始终自生成，永不读取 header
    SetGinRequestID(c, requestID)
    ctx := WithRequestID(c.Request.Context(), requestID)
    c.Request = c.Request.WithContext(ctx)
}
```

[`internal/redisqueue/plugin.go:50`] usage queue 中的 request_id 来源：

```go
requestID := strings.TrimSpace(internallogging.GetRequestID(ctx))
```

全仓库搜索 `X-Request-ID` 和 `x-request-id` —— **零引用**。CLIProxyAPI 完全不读取任何请求 header 中的 request ID。

### 3.3 影响范围

| 受影响的设计节 | 内容 | 影响 |
|---------------|------|------|
| Section 4.1 默认模式 passive | 使用客户端已有的 X-Request-ID | 即使有，CLIProxyAPI 也不会回显到 queue |
| Section 4.2 显式模式 inject_if_missing | MeteringProxy 生成 UUID 注入 | 注入后 CLIProxyAPI 不使用，匹配无效 |
| Section 8.3 入库表 side_usage_events | 按 request_id 关联 | request_id 永远无法匹配 |
| Section 8.4 匹配逻辑 | request_id 精确匹配 | 前提不成立 |
| Section 7.2 token enrichment | side-channel 补齐 token | 无法可靠关联，enrichment 不可行 |
| Section 10.3 Issues | usage_conflict, side_channel_unmatched | 基于无效匹配的分类无意义 |

**结论：Section 4 和 Section 8 的全部精确匹配逻辑在当前 CLIProxyAPI 版本下无法工作。**

### 3.4 修复路径

需要 CLIProxyAPI 侧小改动：在 `gin_logger.go` 的中间件中，优先从 `X-Request-ID` header 读取已有 ID（若存在则使用，否则自生成）。约 5 行代码。

在 CLIProxyAPI 完成此改动之前，MeteringProxy v0.3.0 的 side-channel 精确匹配合并功能不可启用。

## 4. CLIProxyAPI Usage Queue 详细分析

### 4.1 Payload 格式（已确认）

`redisqueue/plugin.go` 中 `queuedUsageDetail` 的 JSON 结构：

| 字段 | 类型 | 来源 | 备注 |
|------|------|------|------|
| `provider` | string | `record.Provider` | 如 "codex", "gemini-cli", "claude" |
| `model` | string | `record.Model` | 上游模型名；空时填充 "unknown" |
| `alias` | string | `record.Alias` | 客户端使用的别名 |
| `endpoint` | string | `internallogging.GetEndpoint(ctx)` | 请求路径 |
| `auth_type` | string | `record.AuthType` | |
| `api_key` | string | `record.APIKey` | **明文** — 设计案 Section 8.3/12.2 要求禁止保存 |
| `request_id` | string | CLIProxyAPI 自生成 | 8 字符随机 hex |
| `tokens.input_tokens` | int64 | `record.Detail.InputTokens` | |
| `tokens.output_tokens` | int64 | `record.Detail.OutputTokens` | |
| `tokens.reasoning_tokens` | int64 | `record.Detail.ReasoningTokens` | |
| `tokens.cached_tokens` | int64 | `record.Detail.CachedTokens` | |
| `tokens.total_tokens` | int64 | 汇总计算 | |
| `latency_ms` | int64 | `record.Latency` | CLIProxyAPI 内部延迟 |
| `source` | string | `record.Source` | |
| `auth_index` | string | `record.AuthIndex` | 关键关联字段 |
| `failed` | bool | 响应状态 ≥ 400 | |
| `timestamp` | time.Time | `record.RequestedAt` | |

与设计案 Section 8.3 的 `side_usage_events` 表结构对比：

- 设计案有 `source_hash` — queue payload 中的 `source` 字段需在入口处 HMAC
- 设计案有 `api_key_hash` — queue payload 中的 `api_key` 是明文，必须即时 hash 不落库
- 设计案有 `error` — queue payload 中无对应字段，需从 `failed` 推导或留空
- 设计案有 `matched_request_usage_id INTEGER` — 指向 `request_usage` 行的整数 ID。但 `request_usage` 表的主关联键是 `request_id`（字符串），跨表关联时建议统一使用 `request_id`

### 4.2 明文 API Key 风险

`plugin.go` 直接将 `record.APIKey` 序列化到 queue payload：

```go
APIKey: apiKey,
```

设计案 Section 12.2 "禁止保存" 列表包含"明文 API key"。实现时必须做到：
1. RESP 消费后立即提取 `api_key` 字段并 HMAC
2. 明文值不在任何日志、内存缓存、错误消息中留存
3. 原始 queue payload 不落库（设计案 Section 8.3 已要求）

### 4.3 队列保留窗口

```go
// queue.go
defaultRetentionSeconds int64 = 60   // 默认
maxRetentionSeconds     int64 = 3600 // 最大
```

设计案 Section 8.1 指出 CLIProxyAPI 侧需配置 `redis-usage-queue-retention-seconds: 300`。这可行（在最大值之内），但需注意：

- MeteringProxy 重启期间，queue 中的 events 会在 retention 超时后被修剪
- 如果 MeteringProxy 消费速度不足（poll_interval=1s, batch_size=50），积压 events 同样会过期
- 设计案已接受此 tradeoff（"队列保留时间有限，MeteringProxy 消费慢或重启时允许丢失 side events"），但这是重要的运维约束

### 4.4 RESP 协议集成

CLIProxyAPI 在同一端口（8317）上通过协议多路复用同时处理 HTTP 和 RESP 连接。RESP 协议使用 management key 进行 AUTH。设计案 Section 8.2 的 `key_file: "/data/management_key"` 配置与此一致。

实现注意：MeteringProxy 的 RESP client 需要处理 CLIProxyAPI 重启导致的连接断开，并按 `reconnect_interval: "5s"` 重连。

## 5. CLIProxyAPI Management API 与配额查询

### 5.1 现有配额端点

[`internal/api/handlers/management/quota.go`] 完整内容：

- `GetSwitchProject` / `PutSwitchProject`
- `GetSwitchPreviewModel` / `PutSwitchPreviewModel`

**仅 toggle，不是配额查询端点。** CLIProxyAPI 没有内置的 provider quota snapshot API。

### 5.2 设计案的配额查询路径

Section 9.2 描述的路径：

```text
MeteringProxy quota poller
  → CLIProxyAPI /v0/management/auth-files     (获取凭证列表)
  → CLIProxyAPI /v0/management/api-call        (代发 HTTP 请求)
  → provider quota endpoint                    (实际查询)
```

是两层串联：先通过 `GET /auth-files` 获取 `auth_index` 列表，再对每个凭证调用 `POST /api-call`。

### 5.3 api-call 机制验证

[`api_tools.go`] 中 `apiCallRequest` 结构：

```go
type apiCallRequest struct {
    AuthIndexSnake  *string           `json:"auth_index"`
    AuthIndexCamel  *string           `json:"authIndex"`
    AuthIndexPascal *string           `json:"AuthIndex"`
    Method          string            `json:"method"`
    URL             string            `json:"url"`
    Header          map[string]string `json:"header"`
    Data            string            `json:"data"`
}
```

关键特性验证：

| 特性 | 状态 | 备注 |
|------|------|------|
| `$TOKEN$` 变量替换 | 支持 | 替换为凭证的 access_token / api_key |
| OAuth token 自动刷新 | 支持 | Gemini OAuth 和 Antigravity OAuth |
| 超时控制 | 默认 60s | 设计案配置 `timeout: "10s"` 需要在 MeteringProxy HTTP client 层额外控制 |
| URL 白名单 | 无 | api-call 可调用任意 URL，MeteringProxy 需自行约束 |
| 每凭证代理设置 | 支持 | 凭证级 proxy_url 优先于全局 proxy-url |

### 5.4 配额查询实施复杂度

设计案要求适配 Claude、Codex、Kimi 三个 provider。每个 provider 需要：
- 各自的 quota 端点 URL
- 各自的认证 header 格式（是否使用 `$TOKEN$`、是否需要额外 header）
- 各自的响应解析逻辑
- 各自的 window_key 提取（5h、7d、monthly 等不同计费周期）

CLIProxyAPI 的 `api-call` 提供了便利通道（代持 token、代发请求），但不提供 provider-specific 的配额解析。这些逻辑全部在 MeteringProxy 侧实现。

### 5.5 auth_index 的作用

[`auth_files.go`] 中的 `ListAuthFiles` 端点返回每个凭证的 `auth_index`，例如：

```json
{
  "provider": "codex",
  "auth_index": "codex:0:sk-atSM...",
  "metadata": { ... }
}
```

设计案中 `quota_snapshots.credential_hash` 需要与这些凭证关联，但两者之间是通过 HMAC 后的值关联的。实现时需注意：
- `auth_index` 本身可能含明文 token 片段（如 `codex:0:sk-atSM...` 中的 `sk-atSM...`）
- HMAC 凭证标识时应使用稳定的、不含 token 的标识符
- 设计案 Section 9.3 使用 `credential_hash` 和 `credential_label_hash` 两个独立 hash——这是正确的设计

## 6. MeteringProxy 当前实现差距

| 设计要求 | 当前 v0.2.0 实现 | 差距评估 |
|----------|-----------------|----------|
| Section 5 两段式 model 扫描（4KB→64KB） | 固定 4KB `io.LimitReader(r.Body, 4096)` | 需可配置的 replay buffer + 分段读取 |
| Section 6 Responses SSE 状态机 | `ExtractResponsesUsage` 仅处理 `response.completed` | 需新建完整状态机 |
| Section 7 来源追踪字段 | 无 `model_returned_source`/`usage_source` | 需新增 DB 列 + event 字段 |
| Section 8 CLIProxyAPI usage queue | 无 | 需全新模块（RESP client + 解析 + 匹配） |
| Section 9 配额快照 | 无 | 需全新模块（poller + provider adapters） |
| Section 4 X-Request-ID 注入 | 仅读取 `requestIDFromHeaders` | 需新增注入逻辑 |
| Section 10 Issues API 扩展 | 已有基础 Issues API | 需扩展 issue class 枚举 |
| Section 11 WebUI | 已有请求表/模型报表/Overview | 需新增来源展示、配额面板、Issues 分组 |

## 7. 设计细节审查

### 7.1 好评决策

1. **Section 3 四层信息模型**：每层字段和规则明确，HTTP 事实层不可被 side-channel 覆盖。与 CLIProxyAPI 实际行为一致（CLIProxyAPI 的 latency 是内部时间线，客户端看到的是 MeteringProxy 转发结果）

2. **Section 6.4 flush 顺序**：`read → write client → flush → parse` 保证了即使解析 panic 也不会影响转发。与当前 `proxy.go` 的 SSE 处理模式一致

3. **Section 16.1 不做模糊匹配**：并发场景下同 endpoint + 同 key + 同 model + 相近时间的请求极常见，自动合并会产生看起来完整但实际错误的数据

4. **Section 7.2 用量合并规则**：HTTP/side-channel 都有 usage 但数值冲突时，保留 HTTP 数据并标记 conflict——正确的保守策略

5. **Section 4.2 显式 mode 是非默认行为**：X-Request-ID 注入改变请求透明度，要求 operator 明确 opt-in

### 7.2 潜在问题

1. **Section 6.2 解析范围中的 `response.incomplete`**：`response.incomplete` 是 OpenAI 的新 event。Section 6.2 列举了它允许读取 `response.incomplete_details.reason`，但 Section 6.3 映射表中没有独立的 `response.incomplete` 行。建议补充映射：`incomplete` 存在时应映射到 `capture_outcome=skipped`、`capture_reason=response_incomplete`、`terminal_reason=incomplete_details.reason`

2. **Section 9.3 `quota_current` 缺少 `credential_label_hash`**：`quota_snapshots` 有但 `quota_current` 没有。如果 WebUI 需要展示凭证标签，current 表需要这个字段

3. **Section 9.4 历史保留仅清理 snapshots**：`side_usage_events` 未提及清理策略。unmatched/expired events 会随运行时间无限累积

4. **Section 8.4 `matched_request_usage_id` 指向 `request_usage` row**：当前 `request_usage` 表的主键是 `id INTEGER PRIMARY KEY AUTOINCREMENT`，存在 `request_id` TEXT 列。如果用整数 ID 关联，需确认跨表 Join 的性能；如果用 `request_id` 关联，不如直接用 `request_id TEXT`

5. **Section 5 扩展扫描性能**：64KB 预读意味着 MeteringProxy 在转发前需等待客户端发送至多 64KB。对于上行带宽受限的客户端（移动网络、卫星链路），这可能增加可感知的延迟

6. **Section 11.2 请求表详情展开**：设计案用 tooltip/展开展示来源信息是好的 UI 策略。但未说明如何展示 side-channel 未启用时的状态——是否显示 "side-channel 未启用" 或完全不显示

## 8. 实施顺序评审

设计案 Section 14 的 6 个 Phase 从依赖关系看是合理的：

```
Phase 1 (Schema) → Phase 2 (SSE) ─┬─→ Phase 6 (WebUI)
               → Phase 3 (Metadata) ─┘
               → Phase 4 (Quota) ────┘
               → Phase 5 (Side-channel) ─┘
```

但有一个关键依赖未标注：**Phase 5 依赖 CLIProxyAPI 侧改动**。在 CLIProxyAPI 支持 X-Request-ID 传播之前，Phase 5 的全部匹配逻辑无法验证。

Phase 4（Quota Snapshot）可独立于 Phase 5 完成——quota 链路不依赖 request-id 匹配。

## 9. 测试覆盖建议

设计案 Section 15 的验收标准覆盖面广。额外建议：

- RESP queue 断连重连的集成测试（模拟 CLIProxyAPI 重启）
- `api-call` 超时测试（验证 MeteringProxy HTTP client 的 `timeout: "10s"` 实际生效）
- 明文 `api_key` 不落库的安全测试（grep 日志、检查 DB 文件、检查 /metering/api/* 响应）
- side event `match_timeout: "10m"` 边界测试（event 恰好超时的情况）
- `inject_if_missing` 模式与已有 `X-Request-ID` 同时存在的去重测试

## 10. 总结

### 设计质量

架构设计层面：优秀。四层观测模型自洽、安全边界清晰、防退化约束严格、来源语义贯穿全文档。Section 16 的决策记录是高质量工程文档的范例，每个 "为什么不做" 都有明确的技术理由。

### 关键阻塞项

**一个阻塞性问题**：Request-ID 传播在设计案假设与 CLIProxyAPI 实际行为之间存在断裂。CLIProxyAPI 不读取 `X-Request-ID` header，其 usage queue 中的 `request_id` 是自生成的随机 ID。这导致 Section 4 和 Section 8 的全部基于 request-id 精确匹配的逻辑无法工作。

此问题需要在实施 Phase 5 之前与 CLIProxyAPI 维护者协调解决。建议将此作为 v0.3.0 启动前的先决条件。

### 建议

1. 在 CLIProxyAPI 侧增加 `X-Request-ID` header 读取支持
2. 或者将 Phase 5（CLIProxyAPI Usage Queue）和依赖它的功能推迟到 v0.4.0，v0.3.0 先交付不依赖 CLIProxyAPI 改动的部分（Phase 1-4 + Phase 6 中不涉及 side-channel 的部分）
3. 无论哪种方案，文档中应明确标注 CLIProxyAPI 依赖版本要求
