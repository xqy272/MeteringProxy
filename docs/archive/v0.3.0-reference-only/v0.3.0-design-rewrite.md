# MeteringProxy v0.3.0 设计方案

版本：v0.3.0
日期：2026-05-05
状态：草案

## 1. 设计目标

v0.3.0 建立 MeteringProxy 的第二条主线——**观测层**——的核心原则：

```text
第一主线：透明转发，不破坏 LLM 体验。
第二主线：建立可信、可解释、可运维的观测层，但不影响第一主线。
```

本版本解决三类实际问题：

1. **Responses API 采集缺失**：当前只捕获 `response.completed` 的 usage；如果流在 completed 之前结束，模型和用量全部丢失。
2. **模型和用量没有来源标注**：`model_returned` 不知道从哪里来的，token 数据不知道是 HTTP 观测还是 side-channel 补齐的。
3. **无法观测 CLIProxyAPI 内部状态**：operator 无法看到哪个凭证被用、哪个快耗尽、side-channel 是否健康。

与现有设计案的关键差异见附录 A。

## 2. 不变量（不谈判）

1. 转发优先于观测完整性。
2. 不存 prompt body、response body、明文 API Key、明文客户端 IP、明文 provider token。
3. 保持请求和响应的透明性：method、path、query、headers、body bytes、status code、SSE bytes 原样转发。
4. 不在转发路径上等待 DB 写入、cost 计算、quota 查询或 side-channel 同步。
5. 队列溢出时丢弃观测事件，不阻塞 LLM 流量。
6. 数据库迁移只做加法，兼容旧版 SQLite。
7. HMAC-SHA256 哈希方案跨部署稳定——salt 文件必须与 DB 一起备份，算法和输出格式不变。

## 3. 信息分层

四层模型，权限递减：

### 3.1 HTTP 事实层

来源：MeteringProxy 转发过程自身。

字段：`created_at`、`request_id`、`endpoint`、`method`、`status`、`latency_ms`、`ttfb_ms`、`stream`、`request_bytes`、`response_bytes`、`api_key_hash`、`client_ip_hash`。

规则：这层是 MeteringProxy 最权威的数据。状态码、延迟、TTFB 永远以 HTTP 事实为准。side-channel 不覆盖这些字段。

### 3.2 协议解析层

来源：对请求前缀、响应 sample、SSE event 的 bounded 解析。

字段：`model_requested`、`model_returned`、`model_returned_source`、`input_tokens`、`output_tokens`、`reasoning_tokens`、`cached_tokens`、`cache_creation_tokens`、`total_tokens`、`usage_source`、`capture_outcome`、`capture_reason`、`terminal_event`、`terminal_reason`、`error_class`、`error_type`、`error_code`、`error_param`。

规则：
- 解析是 best-effort，失败只改 `capture_outcome`/`capture_reason` 或系统计数器。
- 不得影响转发字节。
- 新增字段 `model_returned_source` 和 `usage_source` 标注数据来源，不改变已有 token 数值的计算逻辑。

### 3.3 CLIProxyAPI side-channel 层

来源：CLIProxyAPI Redis-compatible usage queue。

字段：`provider`、`model`、`alias`、`endpoint`、`auth_type`、`auth_index`、`source_hash`、`api_key_hash`、`request_id`、token 细分、`failed`、`latency_ms`、`match_status`。

规则：
- 默认关闭。
- 只做 request-id 精确匹配合并，不做模糊匹配。
- 匹配不到的 side event 单独保存展示，不猜测合并。
- 明文 `api_key`、`source` 入口处立即 HMAC，不落库。

### 3.4 配额快照层

来源：MeteringProxy 后台通过 CLIProxyAPI Management API 只读查询。

配额查询链路完全独立于 LLM 转发：

```text
LLM traffic:  client -> Caddy -> MeteringProxy /v1/* -> CLIProxyAPI /v1/* -> provider
Quota traffic: MeteringProxy background -> CLIProxyAPI /v0/management/* -> provider quota endpoint
```

## 4. Responses SSE 状态机

### 4.1 问题

当前 `ExtractResponsesUsage` 只在 `response.completed` 事件中捕获模型和 usage。如果流在 `response.completed` 之前结束（网络断开、上游错误、`response.failed`），`model_returned` 和所有 token 字段都为空。

### 4.2 设计

每条 Responses SSE 流维护内存态：

```go
type ResponsesStreamState struct {
    SeenCreated    bool
    CreatedModel   string
    SeenCompleted  bool
    CompletedModel string
    HasUsage       bool
    SeenFailed     bool
    SeenError      bool
    UsageSeen      bool
    TerminalEvent  string
    TerminalReason string
}
```

允许解析的事件类型和字段（白名单，不解析不在列表中的字段）：

| 事件类型 | 允许读取的字段 |
|---------|-------------|
| `response.created` | `response.model` |
| `response.completed` | `response.model`、`response.usage`、`response.status` |
| `response.failed` | `response.model`（如有）、`response.status` |
| `response.incomplete` | `response.status`、`response.incomplete_details.reason` |
| `error` | `error.type`、`error.code`、`error.param` |

**禁止**读取或保存：`response.output`、`delta`、`text`、`arguments`、`content`、tool call payload。

### 4.3 结束映射

| 流结束条件 | `capture_outcome` | `capture_reason` | `model_returned_source` | `model_returned` |
|-----------|-----------------|-----------------|----------------------|----------------|
| 有 usage | `captured` | 空 | `usage` | 从 usage 取 |
| completed 无 usage | `skipped` | `response_completed_without_usage` | `response_completed` 或 `response_created` | 从 completed/created 取 |
| 只有 created，无 completed | `skipped` | `stream_ended_without_completed` | `response_created` | 从 created 取 |
| failed/error 事件 | `failed` | `response_error_event` | 能取则取 | 能取则取 |
| SSE 解析错误 | `failed` 或 `skipped` | `parse_error` | — | — |
| 无可识别 event | `skipped` | `usage_not_present` | 空 | 空 |

### 4.4 数据流

流式路径严格保持：

```text
read upstream chunk → write client → flush client → parse copied bytes for observation
```

解析 panic、JSON 错误、行超长、sample overflow 都不能阻止下一次 read/write/flush。

### 4.5 其他 profile 的状态机

- **Chat Completions**：无需状态机，当前 `ExtractChatUsage` 逐行 merge 的方式已足够，因为 Chat Completions 的 usage 行是最终汇总值。
- **Anthropic Messages**：已有 `message_start`（带 model）和 `message_delta`（带 usage）两段式。当前 `ExtractAnthropicUsage` 逐行 merge 已可工作，但可以考虑在 `message_start` 时记录 `model_returned_source=message_start`。本版本暂不扩展 Anthropic 状态机，后续版本按需添加。
- **Gemini**：单次 JSON 或流式 JSON，无 SSE 事件状态需求。

## 5. 模型与用量来源

### 5.1 新增字段

```sql
ALTER TABLE request_usage ADD COLUMN model_returned_source TEXT DEFAULT '';
ALTER TABLE request_usage ADD COLUMN usage_source TEXT DEFAULT '';
ALTER TABLE request_usage ADD COLUMN terminal_event TEXT DEFAULT '';
ALTER TABLE request_usage ADD COLUMN terminal_reason TEXT DEFAULT '';
ALTER TABLE request_usage ADD COLUMN side_usage_event_id INTEGER DEFAULT 0;
```

### 5.2 `model_returned_source` 取值

| 值 | 含义 | 设置时机 |
|---|------|---------|
| `usage` | 来自 response usage 字段 | HTTP 解析到 usage 时 |
| `response_completed` | 来自 response.completed 事件的 model 字段 | completed 有 model 但无 usage |
| `response_created` | 来自 response.created 事件的 model 字段 | 只有 created |
| `http_response` | 来自非流式 response body 的 model 字段 | 非流式路径 |
| `` (空) | 无来源 | 解析失败或 profile 无 model |

effective model 读取规则不变：`model_returned → model_requested → unidentified`。

### 5.3 `usage_source` 取值

| 值 | 含义 |
|---|------|
| `http_response` | MeteringProxy 自身从 HTTP 响应解析到 |
| `cliproxy_side_channel` | 从 CLIProxyAPI usage queue 精确匹配备补 |
| `` (空) | 无 usage 来源 |

**设计决策**：不引入 `usage_confidence` 字段。原因：
- `usage_confidence`（observed/enriched/conflict/missing）与 `usage_source` + `capture_outcome` 高度冗余。
- side-channel 补齐 vs HTTP 观测 vs 冲突的判断完全可以通过 `usage_source` + `side_usage_events.match_status` 在查询时推导。
- 减少一个字段就减少一维 UI 展示复杂度和迁移负担。

### 5.4 `terminal_event` 和 `terminal_reason`

| `terminal_event` | `terminal_reason` | 典型场景 |
|-----------------|-----------------|---------|
| `response.completed` | 空 | 正常完成 |
| `response.completed` | `response_completed_without_usage` | completed 无 usage |
| `stream_end` | `stream_ended_without_completed` | 流在 completed 之前结束 |
| `response.failed` | `response_error_event` | 上游返回 failed 事件 |
| `response.incomplete` | `reason_from_incomplete_details` | 上游返回 incomplete |
| `stream_error` | `parse_error` | SSE JSON 解析错误 |
| `upstream_error` | `http_4xx` / `http_5xx` | 上游返回错误状态码 |

## 6. 有限请求元数据扫描

当前实现读取请求体前 4096 字节提取 `model`。某些请求的 `model` 字段位于更深位置。

### 6.1 两段式扫描

```yaml
request_metadata:
  initial_bytes: 4096
  max_bytes: 65536
  extended_model_scan: true
```

规则：
- 先读 `initial_bytes`（默认 4KB），提取 `model` 和 `stream`。
- 如果 `model` 未找到且 endpoint 是 LLM metered profile，继续读到 `max_bytes`（默认 64KB）。
- 扩展扫描只找顶层 JSON key `"model"`，不递归进入 message/content/tool arguments。
- 找到 `model` 后立即停止扫描。
- 所有已读字节通过 replay reader 原样转发。
- 如果 `model` 在 `max_bytes` 之后，记录为空——这是可接受缺失。

性能上界：并发请求数 × 64KB。1000 并发 = 64MB 额外内存。`extended_model_scan` 可关闭回退到纯 4KB。

### 6.2 实现

将 `internal/proxy/proxy.go` 中的 `bodyPrefix` 读取从固定 `4096` 改为两段：
1. 读 4KB，调用 `topLevelJSONToken(body, "model")`。
2. 如果未找到 model 且 `extended_model_scan=true` 且 profile 是 metered，继续读直到 `max_bytes`。
3. 两段数据合并传给 replay reader。

## 7. CLIProxyAPI usage queue 联动

### 7.1 确认可用的 CLIProxyAPI 能力

基于 CLIProxyAPI 官方文档和源码：

| 能力 | 状态 | 说明 |
|------|------|------|
| Redis RESP 接口（AUTH/LPOP/RPOP） | ✅ 确认 | 同端口 8317，需 management key |
| usage queue payload 字段 | ✅ 确认 | provider, model, alias, endpoint, auth_type, auth_index, source, api_key, request_id, tokens, failed, latency_ms |
| `request_id` 字段 | ⚠️ 需验证 | payload 示例为 `"request_id": "ctx-request-id"`，看起来是 CLIProxyAPI 内部 ID。**需验证客户端 `X-Request-ID` 是否被传播到 queue payload** |
| `/v0/management/auth-files` | ✅ 确认 | 返回凭证列表、状态、auth-index |
| `/v0/management/api-call` | ❌ 未文档化 | 当前 CLIProxyAPI v6.10.7 文档中无此端点 |

### 7.2 配置

```yaml
cliproxy_management:
  enabled: true
  base_url: "http://127.0.0.1:8317/v0/management"
  key_file: "/data/management_key"
  usage_queue:
    enabled: false
    resp_addr: "127.0.0.1:8317"
    queue_name: "queue"
    pop: "LPOP"
    batch_size: 50
    poll_interval: "1s"
    reconnect_interval: "5s"
    match_timeout: "10m"
  quota:
    enabled: false
    providers: ["claude", "codex", "kimi"]
    cache_ttl: "5m"
    timeout: "10s"
    concurrency: 4
    history_retention: "14d"
```

`usage_queue.enabled` 和 `quota.enabled` 默认 false。

### 7.3 请求相关性

**默认模式（passive）**：不新增请求头，只使用客户端已有的 `X-Request-ID` 或 `Request-ID`。

**显式模式（inject_if_missing）**：

```yaml
observability:
  correlation:
    enabled: true
    mode: "inject_if_missing"
    header: "X-Request-ID"
```

- 如果请求已有 `X-Request-ID`，原样保留。
- 如果没有，MeteringProxy 生成 UUID 并添加该 header。
- 生成的 request id 写入 `request_usage.request_id`。
- 这是显式的轻微非透明行为，默认关闭。

**关于 request-id 传播的关键问题**：

如果 CLIProxyAPI 不将客户端传入的 `X-Request-ID` 传播到 usage queue payload 的 `request_id` 字段，则精确匹配无法工作。在此情况下：
1. 首选方案：向 CLIProxyAPI 提交 PR 使其传播 `X-Request-ID`。
2. 降级方案：未匹配的 side event 仍然独立存储和显示，只是无法与 HTTP 请求关联。这仍然有独立价值（可以看到哪些 credential 被使用、哪个 provider 被调用、哪些 model 实际执行）。

### 7.4 入库表

```sql
CREATE TABLE IF NOT EXISTS side_usage_events (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    received_at TEXT NOT NULL,
    received_at_unix INTEGER NOT NULL DEFAULT 0,
    request_id TEXT NOT NULL DEFAULT '',
    matched_request_usage_id INTEGER NOT NULL DEFAULT 0,
    match_status TEXT NOT NULL DEFAULT '',
    provider TEXT NOT NULL DEFAULT '',
    model TEXT NOT NULL DEFAULT '',
    alias TEXT NOT NULL DEFAULT '',
    endpoint TEXT NOT NULL DEFAULT '',
    auth_type TEXT NOT NULL DEFAULT '',
    auth_index TEXT NOT NULL DEFAULT '',
    source_hash TEXT NOT NULL DEFAULT '',
    api_key_hash TEXT NOT NULL DEFAULT '',
    input_tokens INTEGER NOT NULL DEFAULT 0,
    output_tokens INTEGER NOT NULL DEFAULT 0,
    reasoning_tokens INTEGER NOT NULL DEFAULT 0,
    cached_tokens INTEGER NOT NULL DEFAULT 0,
    total_tokens INTEGER NOT NULL DEFAULT 0,
    latency_ms INTEGER NOT NULL DEFAULT 0,
    failed INTEGER NOT NULL DEFAULT 0,
    raw_error TEXT NOT NULL DEFAULT ''
);
```

`match_status` 取值：`matched`、`unmatched`、`conflict`、`duplicate`、`expired`、`invalid_payload`。

`side_usage_events` 没有 `cache_creation_tokens` 列，因为 CLIProxyAPI queue payload 只有 `cached_tokens`（不区分 cache creation 和 cache read）。如果未来 CLIProxyAPI 扩展 payload，再加列。

**禁止保存**：明文 `api_key`、明文 `source`、原始 queue payload。

### 7.5 匹配逻辑

匹配键：`request_id`。

辅助校验：`endpoint roughly matches`、`created_at within match_timeout`、`api_key_hash matches when available`。

`request_id` 为空时禁止自动合并。多条 side event 匹配同一 request 时：第一条有效 usage 可合并，后续标记 `duplicate`。

### 7.6 合并规则

1. HTTP 解析到非零 usage → `usage_source=http_response`。
2. HTTP 无 usage，side-channel 匹配且有非零 usage → 写入 token 字段，`usage_source=cliproxy_side_channel`。
3. HTTP 和 side-channel 都有 usage 且一致 → 保留 HTTP 数据，side event 标记 `matched`。
4. HTTP 和 side-channel 都有 usage 但不一致 → 不覆盖 HTTP，side event 标记 `conflict`。
5. side-channel 无 request-id 匹配 → 不合并，side event 标记 `unmatched`。

**关键**：`capture_outcome` 不因 side-channel enrichment 改变。HTTP SSE 跳过但 side-channel 补齐时，应显示为 "HTTP 采集跳过，side-channel 已补齐"。

## 8. 配额快照

### 8.1 分阶段实现

与现有设计案的关键区别：**将配额拆为两个阶段**，避免阻塞在未确认的 CLIProxyAPI 能力上。

**Phase 4a：凭证健康监控（Credential Health）**

使用 CLIProxyAPI 确认可用的 `/v0/management/auth-files` 端点：

```text
MeteringProxy → GET /v0/management/auth-files → 返回凭证状态列表
```

能获取到的信息：
- 每个凭证的 `provider`、`auth_index`、`status`（ready/unavailable/disabled）
- `email`（需 HMAC 后存储）、`account_type`、`success`/`failed` 计数
- 是否 `disabled` 或 `unavailable`

这些信息虽然不是完整的配额数据，但对运维来说已经很有价值：可以看到哪个凭证挂了、哪个 provider 的账号在频繁失败。

**Phase 4b：配额快照（Quota Snapshot）**

依赖 CLIProxyAPI 提供配额查询能力。实现方式有两种可能：

方案 A — CLIProxyAPI 提供 `/v0/management/api-call` 或等价的配额代理端点：
```text
MeteringProxy → CLIProxyAPI /v0/management/api-call → provider quota endpoint
```
这是理想路径，MeteringProxy 不持有 provider token。

方案 B — MeteringProxy 直接查询 provider 配额端点：
```text
MeteringProxy → CLIProxyAPI /v0/management/auth-files（获取凭证列表）
MeteringProxy → CLIProxyAPI /v0/management/auth-files（获取特定凭证的 OAuth token）
MeteringProxy → provider quota endpoint（使用短期 token）
```
这条路径需要 MeteringProxy 持有临时 token，有更高安全风险。

**本设计推荐方案 A**。如果 CLIProxyAPI 短期不提供 `api-call` 端点，Phase 4b 推迟到 CLIProxyAPI 具备此能力。Phase 4a 独立交付。

### 8.2 Phase 4a 凭证健康入库

```sql
CREATE TABLE IF NOT EXISTS credential_health (
    provider TEXT NOT NULL DEFAULT '',
    credential_hash TEXT NOT NULL DEFAULT '',
    auth_index TEXT NOT NULL DEFAULT '',
    status TEXT NOT NULL DEFAULT '',
    success_count INTEGER NOT NULL DEFAULT 0,
    failed_count INTEGER NOT NULL DEFAULT 0,
    last_checked_at TEXT NOT NULL,
    last_checked_at_unix INTEGER NOT NULL DEFAULT 0,
    last_error TEXT NOT NULL DEFAULT '',
    PRIMARY KEY (provider, credential_hash)
);
```

`status` 取值：`ready`、`unavailable`、`disabled`、`unknown`、`error`。

此表是 Phase 4b `quota_current` 表的前身。当 4b 实现时，可以在同一张表增加 `limit_amount`、`remaining_amount` 等配额列（ALTER TABLE），不需要重建。

### 8.3 Phase 4b 配额快照入库

当 CLIProxyAPI 提供配额查询能力后，扩展为完整快照：

```sql
CREATE TABLE IF NOT EXISTS quota_snapshots (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    checked_at TEXT NOT NULL,
    checked_at_unix INTEGER NOT NULL DEFAULT 0,
    provider TEXT NOT NULL DEFAULT '',
    credential_hash TEXT NOT NULL DEFAULT '',
    plan TEXT NOT NULL DEFAULT '',
    window_key TEXT NOT NULL DEFAULT '',
    limit_amount REAL NOT NULL DEFAULT 0,
    remaining_amount REAL NOT NULL DEFAULT 0,
    used_amount REAL NOT NULL DEFAULT 0,
    unit TEXT NOT NULL DEFAULT '',
    reset_at TEXT NOT NULL DEFAULT '',
    reset_at_unix INTEGER NOT NULL DEFAULT 0,
    status TEXT NOT NULL DEFAULT '',
    error_class TEXT NOT NULL DEFAULT '',
    error_message TEXT NOT NULL DEFAULT '',
    partial INTEGER NOT NULL DEFAULT 0
);
```

### 8.4 查询约束

- quota worker 只由后台定时器、启动预热或 `/metering/api/quota/refresh` 触发。
- `/metering/api/quota/refresh` 只排队 refresh，不同步等待。
- 浏览器永远拿不到 CLIProxyAPI management key 或 provider token。
- 单个 provider 查询失败不影响其他 provider。

## 13. 实施顺序

### Phase 1：Schema 与类型扩展

- 新增配置结构（`request_metadata`、`observability.correlation`、`cliproxy_management`）。
- 新增 DB columns（`model_returned_source`、`usage_source`、`terminal_event`、`terminal_reason`、`side_usage_event_id`）。
- 扩展 `event.Event` 和 `db.UsageRecord`。
- 保持旧 API 字段兼容，空字段按历史数据兼容规则处理。

### Phase 2：Responses SSE 状态机

- 新增 `ResponsesStreamState`。
- 在 stream path 中：read → write → flush → parse for observation。
- 提取 `response.created`/`response.completed` model。
- 记录 `terminal_event`/`terminal_reason`。
- 增加 `model_returned_source` 标注。
- 覆盖测试场景：
  - `response.completed` 带 usage → `captured`
  - `response.completed` 无 usage → `skipped / response_completed_without_usage`
  - 只有 `response.created` → `skipped / stream_ended_without_completed`
  - `response.failed` → `failed / response_error_event`
  - SSE event 拆包、多行 `data:`、无效 JSON、行超长

### Phase 3：请求元数据扫描

- 用 bounded top-level JSON scanner 替换固定 4KB model extraction。
- 保持 replay transparency。
- 增加慢 body、large input、model-after-4KB 测试。

### Phase 4a：凭证健康监控

使用 CLIProxyAPI `/v0/management/auth-files` 端点：
- 新增 CLIProxyAPI management HTTP client。
- 定期轮询 auth-files，HMAC 敏感字段，写入 `credential_health` 表。
- 新增 `/metering/api/quota`（`phase: credential_health`）、`/metering/api/quota/refresh`。
- 新增 `/metering/api/observability` 中 `credential_health` 节点。
- WebUI 凭证健康面板。

### Phase 5：CLIProxyAPI Usage Queue

- 新增 RESP client。
- 消费 LPOP/RPOP。
- 解析 payload 并立即 HMAC 敏感字段（`source`、`api_key`）。
- 实现 request-id 精确匹配。
- 实现 token enrichment 和 conflict 标记。
- 新增 `/metering/api/observability` 中 `side_channel` 节点。

### Phase 4b：配额快照（推迟，依赖 CLIProxyAPI 能力）

- 当 CLIProxyAPI 提供配额代理端点后实施。

## 14. 验收标准

### 14.1 转发透明性

- 所有新增解析测试必须验证响应字节不变。
- SSE 流 flush 顺序保持不变。
- quota 和 usage queue 全部关闭时，代理行为与 v0.2.0 一致。

### 14.2 Responses 采集

必须覆盖：

- `response.completed` 带 usage → `captured`，`model_returned_source=usage`
- `response.completed` 无 usage，带 model → `skipped / response_completed_without_usage`，`model_returned_source=response_completed`
- 只有 `response.created`，无 completed → `skipped / stream_ended_without_completed`，`model_returned_source=response_created`
- `response.failed` → `failed / response_error_event`
- SSE event 拆包、多行 `data:`、无效 JSON、行超长

### 14.3 Side-channel

必须覆盖：

- request-id 精确匹配并补齐 token
- request-id 为空时不合并
- request-id 匹配但 token 冲突
- side event 重复
- queue 断连重连
- invalid payload
- 明文 api_key/source 不落库

### 14.4 凭证健康

必须覆盖：

- auth-files 返回正常
- auth-files 返回空列表
- management key 无效或端点不可达
- 单个 provider 不影响其他 provider
- cache stale
- manual refresh 只排队不阻塞

### 14.5 性能

默认配置下：

- 请求元数据扫描最大 64KB。
- 非流式 response sample 继续受 `max_nonstream_sample_bytes` 限制。
- SSE 单行仍受 max line size 限制。
- usage queue poller 和 credential health poller 不能持有代理热路径锁。
- DB 写入仍通过异步 writer。

## 15. 决策记录

### 15.1 为什么去掉 `usage_confidence` 字段

`usage_confidence`（observed/enriched/conflict/missing）的四值枚举与 `usage_source` + `capture_outcome` + `match_status` 的组合高度冗余：
- `observed` = `usage_source=http_response` + `capture_outcome=captured`
- `enriched` = `usage_source=cliproxy_side_channel` + `capture_outcome=skipped`
- `conflict` = `match_status=conflict`（在 side_usage_events 表）
- `missing` = `usage_source=""` + token 全为 0

去掉一个字段减少迁移负担、DB 宽度、UI 展示复杂度，且所有语义可通过已有字段推导。

### 15.2 为什么将配额拆为 Phase 4a 和 4b

CLIProxyAPI 的 `/v0/management/api-call` 端点在当前文档（v6.10.7）中不存在。将配额快照作为前置条件会阻塞整个实施。Phase 4a 使用确认可用的 `/auth-files` 端点，独立交付凭证健康监控价值。Phase 4b 在 CLIProxyAPI 提供配额代理能力后再实施。

### 15.3 为什么不做模糊匹配

并发下相同 endpoint、相同 API key、相同模型、相近时间的请求非常常见。按时间窗口自动合并会产生看起来完整但实际错误的数据。只允许 request-id 精确匹配。

### 15.4 为什么配额不进请求路径

配额查询慢、不稳定、provider-specific，可能触发额外限制。MeteringProxy 职责是观测，不是调度。

### 15.5 为什么 side-channel 不覆盖 HTTP 事实

CLIProxyAPI 内部视角能解释 provider、账号和 usage，但客户端实际看到的状态码、延迟、TTFB 仍以 MeteringProxy 转发为准。两者是不同层级的事实。
