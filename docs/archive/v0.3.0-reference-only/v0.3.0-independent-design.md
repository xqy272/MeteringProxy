# MeteringProxy v0.3.0 独立设计方案

日期：2026-05-05
状态：草案
作者：Claude（基于 CLIProxyAPI + MeteringProxy 代码库审查）

## 1. 设计动机

当前 v0.2.0 的 `request_usage` 表里，看一行数据时无法回答三个基本问题：

- **模型名是从哪来的？** 请求体里读到的？响应里返回的？还是根本没读到？
- **token 数是怎么来的？** HTTP 响应解析的？还是根本没解析到？
- **为什么有些行有模型没 token，有些行有 token 没模型？**

v0.3.0 解决一件事：**让每一条观测数据都带着来源标记——可信、可解释。**

## 2. 版本边界

### 2.1 v0.3.0 交付

1. 来源追踪字段：`model_returned_source`、`usage_source`
2. Responses SSE 状态机：拆分模型观察和 usage 观察
3. 两段式模型扫描：默认 4KB，可扩展至 64KB
4. `terminal_event` / `terminal_reason`：记录 SSE 流结束方式
5. API 与 WebUI 适配

### 2.2 v0.3.0 不做

- **不做 CLIProxyAPI usage queue 消费** — request-ID 传播链路在 CLIProxyAPI 当前版本下不工作（见审查报告第 3 节）
- **不做配额快照** — 独立功能模块，CLIProxyAPI 无现成配额端点，与核心目标无直接依赖
- **不做 X-Request-ID 注入** — 无 CLIProxyAPI 侧配合时为无效操作
- **不新增任何表** — `side_usage_events`、`quota_snapshots`、`quota_current` 均不创建
- **不做 Issues API 扩展** — 新 issue class 依赖不存在的数据源

## 3. 新增字段与迁移

```sql
-- Schema v6
ALTER TABLE request_usage ADD COLUMN model_returned_source TEXT DEFAULT '';
ALTER TABLE request_usage ADD COLUMN usage_source TEXT DEFAULT '';
ALTER TABLE request_usage ADD COLUMN terminal_event TEXT DEFAULT '';
ALTER TABLE request_usage ADD COLUMN terminal_reason TEXT DEFAULT '';

CREATE INDEX IF NOT EXISTS idx_request_usage_model_returned_source
ON request_usage(model_returned_source, created_at_unix);

CREATE INDEX IF NOT EXISTS idx_request_usage_usage_source
ON request_usage(usage_source, created_at_unix);
```

不新增表。增量迁移，不回填历史数据。

### 3.1 model_returned_source

| 值 | 含义 | 触发条件 |
|----|------|----------|
| `usage` | 来自 response usage 字段 | Chat/Anthropic/Gemini 的 usage chunk；Responses completed.usage |
| `response_completed` | 来自 completed event | Responses response.completed 有 model 但无 usage |
| `response_created` | 来自 created event | Responses response.created 有 model，无后续 event |
| `response_failed` | 来自 failed event | Responses response.failed 有 model |
| (空) | 历史数据 / 未识别 | v0.2.0 数据或完全不适用 |

### 3.2 usage_source

| 值 | 含义 |
|----|------|
| `http_response` | MeteringProxy 从 HTTP 响应解析 |
| (空) | 未解析到 / 历史数据 |

v0.3.0 永远为 `http_response` 或空。`cliproxy_side_channel` 留给后续版本。

### 3.3 terminal_event / terminal_reason

| terminal_event | terminal_reason | 场景 |
|----------------|-----------------|------|
| `response.completed` | (空) | 正常 completed |
| `response.failed` | `response_error_event` | 上游返回 failed |
| `response.incomplete` | `incomplete_details.reason` 的值 | 上游返回 incomplete |
| `error` | (空) | SSE error event |
| `stream_end` | `stream_ended_without_completed` | 流在 completed 前结束 |
| `stream_end` | `client_disconnect` | context canceled |
| (空) | (空) | 非流式 |

### 3.4 历史数据兼容

- WebUI 对空 `model_returned_source` 且 `model_returned` 非空 → 显示 "来自响应 (v0.2)"
- WebUI 对空 `usage_source` 且 token > 0 → 显示 "HTTP 响应"
- API 直接返回空字符串，前端做兼容展示

## 4. Responses SSE 状态机

### 4.1 当前问题

`ExtractResponsesUsage` 只处理 `response.completed` 且必须带 `usage`。实际场景：

- completed 存在无 usage → **模型丢了**
- 只有 created 就断流 → **模型丢了**
- failed / error / incomplete → **全部丢失**

### 4.2 状态结构

```go
type ResponsesStreamState struct {
    SeenCreated      bool
    CreatedModel     string

    SeenCompleted    bool
    CompletedModel   string
    CompletedHasUsage bool
    Usage            *extractor.UsageInfo

    SeenFailed       bool
    FailedModel      string

    SeenErrorEvent   bool
    ErrorType        string
    ErrorCode        string
    ErrorParam       string

    SeenIncomplete   bool
    IncompleteReason string

    ParseErrors      int64
}
```

### 4.3 解析白名单

| Event | 读取 | 禁止 |
|-------|------|------|
| `response.created` | `response.model` | `response.output` 全部 |
| `response.completed` | `response.model`, `response.usage`, `response.status` | `response.output` |
| `response.failed` | `response.model` | response body |
| `response.incomplete` | `response.model`, `response.incomplete_details.reason` | `response.output` |
| `error` | `type`, `code`, `param` | `message`（可能含用户内容） |

只解析顶层 JSON key。

### 4.4 结束映射

```
usage 非空:
  capture_outcome         = captured
  model_returned          = usage.Model
  model_returned_source   = "usage"
  usage_source            = "http_response"
  terminal_event          = "response.completed"

completed 存在, 无 usage:
  capture_outcome         = skipped
  capture_reason          = "response_completed_without_usage"
  model_returned          = completed.Model ?? created.Model
  model_returned_source   = "response_completed"
  terminal_event          = "response.completed"

created 存在, 无 completed/failed/incomplete:
  capture_outcome         = skipped
  capture_reason          = "stream_ended_without_completed"
  model_returned          = created.Model
  model_returned_source   = "response_created"
  terminal_event          = "stream_end"

failed 存在:
  capture_outcome         = failed
  capture_reason          = "response_error_event"
  model_returned          = failed.Model ?? created.Model
  model_returned_source   = "response_failed"
  terminal_event          = "response.failed"

error event:
  capture_outcome         = failed
  capture_reason          = "response_error_event"
  terminal_event          = "error"
  (记录 error_class/type/code，不存 message)

incomplete 存在:
  capture_outcome         = skipped
  capture_reason          = "response_incomplete"
  model_returned          = incomplete.Model ?? created.Model
  model_returned_source   = "response_completed"
  terminal_event          = "response.incomplete"
  terminal_reason         = incomplete.Reason

无任何可识别 event:
  capture_outcome         = skipped
  capture_reason          = "usage_not_present"
  model_returned          = (空)
```

### 4.5 flush 顺序

与当前 `proxy.go` 一致，保持不变：

```text
read upstream chunk → write client → flush client → parse copied bytes
```

任何解析错误不阻断下一次 read/write/flush。

### 4.6 其他 API

Chat Completions、Anthropic Messages、Gemini 的 SSE 解析不变。它们的 model 和 usage 在同一 chunk 出现，逐行 merge 已足够。对于它们：

- `model_returned_source` = `"usage"`
- `usage_source` = `"http_response"`
- `terminal_event` / `terminal_reason` 留空

## 5. 两段式模型扫描

### 5.1 当前行为

```go
bodyPrefix, _ = io.ReadAll(io.LimitReader(r.Body, 4096))  // 固定 4KB
```

### 5.2 改进

```
1. 读 initial_bytes（默认 4KB），replayReader 缓存
2. 在 initial_bytes 中找顶层 "model"
3. 找到 → 停止
4. 没找到 && max_bytes > 0 && endpoint 是 LLM metered profile：
   继续读到 max_bytes 或 EOF
   只解析顶层 JSON key（不进 messages/content/tool_calls）
   找到即停
5. 已读字节通过 replayReader 原样转发
```

边界条件：
- 遇到嵌套对象/数组 → skip 整个结构，继续后续顶层 key
- model 在 max_bytes 之后 → `model_requested=""`，可接受缺失
- 慢客户端 → 增加转发前等待，可通过配置关闭
- 读取失败 → 已读字节正常 replay

### 5.3 配置

```yaml
request_metadata:
  initial_bytes: 4096
  max_bytes: 65536
  extended_model_scan: true   # false = v0.2.0 行为
```

## 9. 实施顺序

```

  - 4 个新列 + 2 个索引

```

Phase 2-4 可并行。

## 10. 验收标准

### 10.1 转发透明性

- 所有新解析路径下响应字节与 v0.2.0 一致
- SSE flush 顺序不变
- `extended_model_scan: false` 时行为与 v0.2.0 完全一致

### 10.2 Responses 采集

| 场景 | capture_outcome | capture_reason | model_returned_source |
|------|-----------------|----------------|----------------------|
| completed 带 usage | captured | — | usage |
| completed 无 usage | skipped | response_completed_without_usage | response_completed |
| created only，断流 | skipped | stream_ended_without_completed | response_created |
| failed | failed | response_error_event | response_failed |
| error event | failed | response_error_event | (能取则取) |
| incomplete | skipped | response_incomplete | response_completed |
| SSE 拆包 | (按最终状态) | (按最终状态) | (按最终状态) |
| 无效 JSON | 不影响 | parse_errors counter++ | — |

### 10.3 模型扫描

- model 在 4KB 内 → 正确
- model 在 4-64KB → 正确
- model 在 64KB 后 → `""`，可接受
- 嵌套对象 → 正确 skip

### 10.4 兼容性

- 旧数据新字段为空 → API 返回 `""`
- WebUI 兼容文案
- 无 CLIProxyAPI 依赖

## 11. 决策记录

### 为什么 v0.3.0 不做 CLIProxyAPI 集成

CLIProxyAPI 的 `logging/requestid.go` 自生成随机 ID，`gin_logger.go` 不读取 `X-Request-ID` header。usage queue 中的 `request_id` 是 CLIProxyAPI 内部 ID，与 MeteringProxy 可设置的任何 header 无关。精确匹配前提不成立。不做模糊匹配（并发下张冠李戴）。

### 为什么不加 `usage_confidence`

v0.3.0 只有一种可产生的值（`observed`）。其余三个值全部依赖 side-channel。加入单值字段无信息增益。后续版本中 `usage_source` + `capture_outcome` + `side_usage_events.match_status` 组合已可推导等效语义。

### 为什么配额推迟

独立功能模块，CLIProxyAPI 无现成配额查询端点（`quota.go` 仅含 toggle），与"观测数据可信"核心目标无直接依赖。
