# v0.3.0 设计案 Review

审查对象：`docs/v0.3.0-observability-design.md`
日期：2026-05-05
审查基线：MeteringProxy 当前代码库（schema v5）+ CLIProxyAPI v6.10.7 官方文档

---

## 1. 总体评价

设计案结构完整、层次清晰，四层信息模型（HTTP 事实 → 协议解析 → Side Channel → 配额快照）的抽象合理。核心不变量（转发优先、不存明文、不覆盖事实）与项目 CLAUDE.md 一致。以下按模块逐一分析关键风险和改进建议。

---

## 2. Responses SSE 状态机（Section 6）

**风险：中**

**当前缺口**：`internal/proxy/proxy.go` 的 Responses 流式处理只通过 `ExtractResponsesUsage` 在每条 SSE 行上尝试提取 `response.completed` 中的 usage，没有跟踪 `response.created`/`response.failed`/`response.incomplete`，也没有记录 `terminal_event`。

**设计合理性**：状态机拆分"模型观察"和"usage 观察"正确。`seen_response_created`/`completed_model`/`usage_seen` 等内存态字段设计合理。

**集成风险**：

- 当前 `streamproto.OpenAISSE` 只定义了 `CompletionMarker` 和 `MaxLineSize`，没有针对 Responses 事件类型的子协议。状态机需要与 `StreamExtractor` 接口协调——建议在 `profile.EndpointProfile` 中添加 `StreamStateMachine` 接口，由 Responses profile 实现具体状态转移，`proxy.handleStream` 在 flush 后调用状态机更新。

- Section 6.2 列出了"允许读取的字段"白名单，但需确保实现时不意外读取 `response.output` 或 `delta` 等禁止字段。建议代码中用白名单 struct 或 JSON tag 过滤来强制约束。

**建议**：为 Responses SSE 创建独立的 `StreamProtocol` 实现（而非复用 `OpenAISSE`），因为 Responses 的事件类型（`response.created`、`response.completed`、`response.failed`、`response.incomplete`）与 Chat Completions 的 `data: {"choices": ...}` 格式完全不同。

## 3. CLIProxyAPI Usage Queue 联动（Section 8）

**风险：中高**

### 3.1 与 CLIProxyAPI 实际能力对齐

| 设计描述 | CLIProxyAPI 实际 | 评估 |
|---------|-----------------|------|
| RESP 协议 AUTH/LPOP/RPOP | 文档确认支持 | ✅ 匹配 |
| `queue_name: "queue"` | key 参数被忽略，用 `queue` 可读即可 | ✅ 匹配 |
| `batch_size: 50` | LPOP 支持 count 参数 | ✅ 匹配 |
| payload 字段: provider, model, alias, endpoint, auth_type, auth_index, source, api_key, request_id, tokens, failed, latency_ms | 完全匹配 CLIProxyAPI 文档 | ✅ 匹配 |
| 明文 source/api_key 入口 HMAC | CLIProxyAPI 发送明文 `source`（邮箱）和 `api_key` | ✅ 设计正确处理 |

### 3.2 request-id 关联是最大的不确定性

设计 Section 4 提到 CLIProxyAPI "usage queue 当前会输出其内部 request id"，而设计选择 `X-Request-ID` 作为关联 header。但 CLIProxyAPI usage queue payload 中 `request_id` 字段示例值为 `ctx-request-id` 格式，看起来是 CLIProxyAPI 内部生成的 context ID。

如果 CLIProxyAPI 不把客户端传入的 `X-Request-ID` 传播到 usage queue 的 `request_id` 字段，则精确匹配将无法工作。

**建议**：

1. 在实施 Phase 5 之前，验证 CLIProxyAPI 当请求携带 `X-Request-ID` 时是否在 usage queue 输出中使用该值。
2. 如果不是，向 CLIProxyAPI 提交 issue/PR 让其传播 `X-Request-ID`。
3. 准备降级方案：未匹配的 side event 仍然独立存储展示（可以看到哪个 credential 被使用、哪个 provider 被调用），只是无法与 HTTP 请求关联。这仍然有独立价值。

### 3.3 Queue 保留时间

设计建议 `redis-usage-queue-retention-seconds: 300`（5分钟），但 CLIProxyAPI 默认 60 秒，最大 3600 秒。如果 MeteringProxy 的 `poll_interval` 是 1s 且 `batch_size` 是 50，在 60 秒保留期内理论上是够用的。但需要在配置文档中明确要求将 `redis-usage-queue-retention-seconds` 设为至少 300。

### 3.4 `cache_creation_tokens` 字段

CLIProxyAPI queue payload 只有 `cached_tokens`，没有区分 cache creation 和 cache read。Anthropic 的 `cache_creation_input_tokens` 和 `cache_read_input_tokens` 是两个不同的计费维度。`side_usage_events` 表的 `cache_creation_tokens` 列从 CLIProxyAPI side-channel 可能永远填不进值，需文档注明或改为可空。

## 4. 配额快照（Section 9）

**风险：高 — 这是 v0.3.0 中风险最高的模块**

设计描述的配额查询路径：

```text
MeteringProxy quota poller
  -> CLIProxyAPI /v0/management/auth-files
  -> CLIProxyAPI /v0/management/api-call
  -> provider quota/usage endpoint
```

**核心问题：`/v0/management/api-call` 端点在 CLIProxyAPI 当前文档（v6.10.7）中不存在。** 现有 Management API 没有提供一个通用的 provider quota 代理端点。`/v0/management/auth-files` 只返回凭证列表和状态，不包含配额信息。

这意味着：

1. 如果 CLIProxyAPI 未来添加此端点：MeteringProxy 需要等待 CLIProxyAPI 更新。设计文档应明确标注这个依赖关系和阻塞状态。

2. 替代方案：部分第三方工具（Quotio、Quota Inspector）通过直接调用 provider quota 端点来查询配额。但设计明确禁止 MeteringProxy 直接持有 provider token 或调用 provider 端点。这些第三方工具实际是通过 CLIProxyAPI 的 management API 获取凭证信息后自行查询 provider。

**建议**：

- 在 Phase 4 开始前，验证 CLIProxyAPI 是否有未文档化的 `api-call` 功能，或确认其路线图上有此计划。
- 准备降级方案：如果 `api-call` 不可用，Phase 4 可以只实现基于 `/v0/management/auth-files` 的凭证健康监控（在线/离线状态、success/failed 计数），作为配额信息的替代来源。
- 考虑部分配额信息可以从 `auth-files` 的 `status` 字段推断（`ready`/`unavailable` 等凭证状态）。

## 5. 请求元数据扫描（Section 5）

**风险：低**

两段式扫描设计合理。当前代码在 `proxy.go` 中读取固定 4096 字节前缀。扩展到 64KB 对大多数请求影响微小（JSON 中 `model` 通常在前几 KB），但对慢客户端场景有明显影响。

**建议**：

- 实现 `max_bytes` 阶梯式读取而非一次性分配 64KB buffer。先用 4KB，找 model 时立即停止；找不到时再读下一块。
- 阶梯式读取的额外内存开销 = 并发数 × max_bytes，对于 1000 并发是 64MB。需在文档中注明。

## 6. 数据模型与迁移（Section 3, 13）

**风险：低**

新增 6 列 + 3 表 + 6 索引的增量迁移方案与项目现有迁移风格一致（`ensureColumns` + `schema_tasks`）。

### 6.1 `side_usage_event_id` 列

设计在 `request_usage` 中添加 `side_usage_event_id INTEGER DEFAULT 0` 用于精确匹配关联。这是合理的反规范化，避免查询时 JOIN。

### 6.2 `quota_current` 的 PRIMARY KEY

`quota_current` 的主键是 `(provider, credential_hash, window_key)`。如果同一个 credential 在同一个 window 下有多条快照（例如刷新间隔内数据变化），需确保 UPSERT 逻辑正确。设计文档中只描述了 INSERT 到 `quota_snapshots`（历史表）和 UPDATE `quota_current`（当前表），逻辑合理。

### 6.3 缺失索引

`side_usage_events` 上有 `(request_id)` 和 `(match_status, received_at_unix)` 索引，但缺少 `(matched_request_usage_id)` 索引。如果需要从 side event 反查 request_usage，可能需要此索引。

## 7. API 设计（Section 10）

**风险：低**

新增端点设计合理。`/metering/api/quota/refresh` 只排队不同步等待——正确的做法，避免了 N+1 provider 查询阻塞 HTTP handler。

`/metering/api/observability` 端点汇总三项健康状态，设计简洁。

**建议**：`/metering/api/requests` 返回的 `side_usage_event_id` 对前端关联查询有用，但应确保此字段在有 side-channel 数据但未匹配时为 0（而非 NULL），与 DEFAULT 0 一致。

---

## 8. 安全与隐私（Section 12）

**评估：优秀**

设计明确列出了允许保存和禁止保存的字段，与 CLAUDE.md invariant #2 一致。特别注意：

- `quota_snapshots.error_message`：系统生成的错误（网络超时、解析失败等），不是 provider 返回的用户内容。与 CLAUDE.md 中 "error_message DB column is reserved for future system-generated diagnostic messages" 一致。✅
- 明文 `source`/`api_key` 在入口处 HMAC 后丢弃：与 CLIProxyAPI queue payload 中的明文字段处理方式匹配。✅

---

## 9. `usage_confidence` 字段

**风险：低 — 建议去掉**

`usage_confidence`（observed/enriched/conflict/missing）的四值枚举与 `usage_source`（http_response/cliproxy_side_channel/""）加上 `capture_outcome` 和 `match_status` 的组合高度冗余：

| `usage_confidence` 值 | 等价组合 |
|----------------------|---------|
| `observed` | `usage_source=http_response` + `capture_outcome=captured` |
| `enriched` | `usage_source=cliproxy_side_channel` + `capture_outcome=skipped` |
| `conflict` | `match_status=conflict`（在 `side_usage_events` 表） |
| `missing` | `usage_source=""` + token 全为 0 |

去掉一个字段减少迁移负担、DB 宽度、UI 展示复杂度，且所有语义可通过已有字段推导。建议不新增此字段。

## 10. 实施顺序调整建议

设计文档中 Phase 4 (Quota) 在 Phase 5 (Usage Queue) 之前。建议调换顺序：

| 原顺序 | 建议顺序 | 理由 |
|--------|---------|------|
| Phase 4: Quota | Phase 4a: 凭证健康 + Phase 5: Usage Queue | Usage Queue 只依赖 RESP 协议（已确认可用），Quota 全功能依赖未确认的 CLIProxyAPI 端点 |
| Phase 5: Usage Queue | Phase 4b: Quota Snapshot（推迟到依赖确认后） | 避免被外部依赖阻塞 |

进一步建议将 Quota 拆为两个 Phase：

- **Phase 4a**：基于 `/v0/management/auth-files` 的凭证健康监控，只使用确认可用的 API。
- **Phase 4b**：完整配额快照，依赖 CLIProxyAPI 提供 `/api-call` 或等价能力后再实施。

## 11. 设计文档中值得补充的细节

### 11.1 SSE 事件解析安全边界

Section 6.2 列出了允许读取的 JSON 字段，但应补充单条 SSE event 的 JSON 解析大小上界（例如 max 1MB per event），防止恶意上游注入超大 JSON 导致解析 panic 或 OOM。

### 11.2 Usage Queue 消费幂等性

如果 MeteringProxy 重启，LPOP 已消费但未写入 DB 的 events 会丢失。设计应明确：queue 事件丢失是可接受的（与 invariant #1 一致），不需要 WAL 或两阶段提交。

### 11.3 Quota poller 失败重试策略

Section 9.2 说"单个 provider 失败不影响其他 provider"，但没有描述重试策略。建议补充指数退避或固定间隔重试，并设置最大连续失败次数。

### 11.4 `credential_label_hash` 的处理

`quota_snapshots` 中有 `credential_label_hash`，但 CLIProxyAPI `auth-files` API 返回的 `label` 是可选字段。如果 label 为空，hash 应为空字符串而非空字符串的 hash。
