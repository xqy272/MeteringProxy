# v0.5.0 Migration and Rollback Rehearsal

Date: 2026-07-22 (Asia/Shanghai)

## Scope

This rehearsal verifies the additive v0.5.0 Key index against a database created by the actual v0.4.2 code and startup path. It covers:

1. normal v0.4.2 -> v0.5.0 upgrade;
2. forced process termination while SQLite is writing the new index transaction, followed by recovery;
3. v0.4.2 binary + v0.4.2 pricing rollback after the database has been opened by v0.5.0.

No production database, salt, configuration, server, or provider was accessed. The fixture is synthetic and lived under `C:\tmp`; repository-local ignored SQLite, salt, binary, and development config files were not touched.

## Fixture

The v0.4.2 binary first created and bound the database to a synthetic salt. The fixture generator then used the same distribution as the million-row Key benchmark:

| Property | Value |
|---|---:|
| `request_usage` rows | 1,000,000 |
| distinct Key hashes | 100 |
| distinct returned models | 100 |
| time span | 2026-06-22 00:00:00Z through 2026-07-22 00:00:00Z |
| request_usage indexes at v0.4.2 baseline | 14 |
| new composite index at baseline | absent |
| checkpointed DB bytes | 635,883,520 |
| baseline SHA-256 | `A7719E16863AEADEA26ECEE0E6F5A2A1289D35AC0588B9EA91E677968C320E99` |

Three byte-identical copies of that checkpointed baseline were used so the scenarios could not influence one another.

## Results

### Normal upgrade and idempotent reopen

The v0.5.0 binary reached its startup self-check in 1,544 ms on the first open.

| Check | Result |
|---|---|
| `PRAGMA integrity_check` | `ok` |
| row count after upgrade | 1,000,000 |
| index columns | `api_key_hash, created_at_unix` |
| request_usage index count | 15 |
| DB bytes after checkpoint | 716,267,520 |
| DB growth | 80,384,000 bytes (12.64% of baseline) |

A second v0.5.0 open reached the self-check in 94 ms. The DB size did not change, the row count remained 1,000,000, and the index remained present. This verifies the `CREATE INDEX IF NOT EXISTS` path is idempotent for the exercised database.

### Interrupted index construction and recovery

The v0.5.0 process was force-terminated 698 ms after launch, after the new SQLite WAL had grown to 5,397,232 bytes. This places the interruption inside the large index transaction rather than before migration began.

Independent SQLite recovery immediately after termination produced:

| Check | Result |
|---|---|
| `PRAGMA integrity_check` | `ok` |
| row count | 1,000,000 |
| composite index | absent (the uncommitted DDL was rolled back atomically) |
| checkpointed DB bytes | 635,883,520 (baseline size) |

Restarting the same v0.5.0 binary then reached its self-check in 1,575 ms. The composite index was present with the expected two columns, the row count remained 1,000,000, `integrity_check` remained `ok`, and the final checkpointed size was 716,267,520 bytes.

### Old-bundle rollback

A fresh baseline copy was first upgraded by v0.5.0 in 1,556 ms. The v0.4.2 binary was then started against that upgraded database with the v0.4.2 `pricing.yaml`, the same salt, and a v0.4.2-compatible config.

| Check | Result |
|---|---|
| v0.4.2 startup self-check | pass in 84 ms |
| pricing loaded by v0.4.2 | tag-matching v0.4.2 `pricing.yaml` |
| `PRAGMA integrity_check` | `ok` |
| row count | 1,000,000 |
| `PRAGMA user_version` | 8 |
| composite index | retained and safely ignored by v0.4.2 |

This validates the documented rollback contract: the additive index does not block the old binary, but rollback must restore the old binary/image together with its matching old config and pricing file. The salt must remain paired with the database.

## Release interpretation

- The exercised migration is atomic, restartable, and backward-readable at the v0.4.2 boundary.
- The measured 1.5-second migration and 12.64% file growth apply only to this synthetic fixture and host. They are not a production SLO and must not be extrapolated linearly.
- Production operators must still stop the writer, preserve a coherent DB/WAL/SHM + salt + config + pricing snapshot, and reserve a maintenance window before first v0.5.0 startup.
- Production deployment remains outside this release rehearsal and outside the v0.5.0 construction scope.
