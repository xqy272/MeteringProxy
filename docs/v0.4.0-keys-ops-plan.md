# v0.4.0 Keys Ops + Pricing Refresh

> **已废弃草稿：** 本文件仅保留决策历史，已由 `v0.5.0-key-insights-plan.md` 取代，不得作为施工或定价依据。
>
> 目标版本规划（只读观测 + 软告警 + 定价刷新；不拦截、不改 LLM 语义）

## 一句话目标

把客户端 API Key 做成可运营的只读观测维度（用量 / 成本 / 下钻 / 软告警），并同步补齐本批模型定价；全程不进代理热路径、不拦截、不改 LLM 语义。

适用场景：朋友共享中转、按 key 看用量与成本、超预算时收到告警并由人工处理。

## 2. 对象说明

本版「Key」仅指：

- **客户端调用中转时使用的 API Key**
- 存储形态：`request_usage.api_key_hash`（HMAC-SHA256，64 位小写 hex）

不是：

- CPA 上游 provider 凭据
- CPA management key
- `side_usage_events.api_key_hash`（命名空间不同，禁止当同一实体对齐）

提取来源（已有，本版不改行为）：

1. `Authorization: Bearer <token>`
2. `X-API-Key`
3. `X-Goog-API-Key`
4. 查询参数 `?key=`（Gemini）

空 key 在聚合中归为 `unknown`。

## 3. 产品形态

### 3.1 日常使用

1. 给朋友单独一把 API key（签发在 CPA / 上游侧，本项目不负责签发）
2. 本地或配置中记录：`friend-a` ↔ full/short hash
3. WebUI **Keys** Tab 查看：昵称、请求数、失败率、token、**成本**
4. 点击某 key → 查看该 key 的请求明细与错误

### 3.2 告警

- 配置 `daily_usd` / `daily_tokens`
- 达到 `warn_ratio`（默认 80%）→ warning
- 达到 100% → exceeded
- 出现在 Issues；可选 webhook 通知
- **只告警，不拦截**

### 3.3 定价

- `ds-openai-pro` / `ds-openai-flash` 自动映射 DeepSeek V4
- `grok-4.5` 可估价（含 long context 档，阈值待确认）
- `grok-imagine-image-quality` 在能拿到 `image_count` / `size` 时尽量按张估价

## 4. 功能规格

### A. Keys 观测增强（核心）

#### A1. `GET /api/keys` enriched

在现有字段上增加成本、失败率、标签等：

```json
{
  "key_hash": "…",
  "label": "friend-a",
  "request_count": 120,
  "failed_count": 3,
  "failure_rate": 0.025,
  "input_tokens": 100000,
  "output_tokens": 50000,
  "total_tokens": 150000,
  "estimated_cost": 1.23,
  "cost_known": true,
  "unpriced_models": 0,
  "latest_seen_at": "2026-07-22T12:00:00Z"
}
```

实现要点：

- 复用 models 路径的成本计算（`CostWithCacheCreation`）
- **按 key → 再按 model 聚合后算价**（不能对 total_tokens 盲算）
- 纯 dashboard / 读路径，不碰 proxy 热路径

涉及文件（预期）：

- `internal/db/db.go`：`Keys` 扩展或新增 `KeyModels`
- `internal/event/report.go`：`KeyReport`
- `internal/webui/server.go`：`handleKeys`
- `internal/webui/static/*`：Keys 表增加 Cost / Label

#### A2. 按 Key 过滤请求

```http
GET /api/requests?range=24h&key_hash=<64hex|prefix>
```

- 支持完整 hash
- 可选支持唯一前缀（建议 ≥8 且唯一，否则 400）
- `store.ReportStore.Requests` 增加 `keyHash` 参数
- UI：Keys 行可点 → Requests 带 filter

#### A3. Issues 与 Key 可读性

- 已有聚合键含 `api_key_hash`（保持）
- UI 显示 `label || shortHash`
- 可选 `GET /api/issues?key_hash=…` 过滤（建议同做）

#### A4. `key_labels` 配置

```yaml
key_labels:
  # full 64-char api_key_hash -> display name
  "a1b2c3d4e5f6…": "friend-a"
  "112233445566…": "self"
```

规则：

- 只接受 **完整 hash → 昵称**
- 不接受明文 key
- 缺省不影响计量
- API / UI 仅多一个 `label` 字段

本版 **不做** 在线「明文 key → hash」预览接口（WebUI 若无鉴权，存在把明文送进浏览器的风险）。对照方式写进 README：看首条流量后的 short hash，或在受控环境用同一 salt 离线计算。

---

### B. 软预算告警（同版，零拦截）

#### B1. 配置

```yaml
key_budgets:
  enabled: true
  # 窗口：滚动 24h（默认建议）
  window: rolling_24h
  default:
    daily_usd: 0          # 0 = 禁用
    daily_tokens: 0
    warn_ratio: 0.8
  keys:
    "a1b2c3d4e5f6…":
      daily_usd: 5
      daily_tokens: 2000000
  webhook_url: ""         # 可选
  check_interval: 5m
```

#### B2. 后台 checker

- 独立 goroutine，**只读 DB**
- 默认窗口：**滚动 24h**（实现简单、跨时区坑少；文档写清）
- 产出：
  - Issues 系统项：`key_budget_warning` / `key_budget_exceeded`
  - 可选 webhook JSON
- **永不**参与 proxy 拒绝逻辑
- checker 失败 / DB 错误 → fail-open，不影响 LLM 流量

#### B3. WebUI

- Keys 行显示预算进度（若已配置）
- Issues 展示超限类系统项

### C. 定价刷新

#### C1. DeepSeek 别名（配置即可）

```yaml
deepseek-v4-flash:
  input_per_1m: 0.14
  cached_input_per_1m: 0.0028
  output_per_1m: 0.28
  aliases:
    - ds-openai-flash

deepseek-v4-pro:
  input_per_1m: 0.435
  cached_input_per_1m: 0.003625
  output_per_1m: 0.87
  aliases:
    - ds-openai-pro
```

说明：`ds-openai-pro` = DeepSeek V4 Pro；`ds-openai-flash` = DeepSeek V4 Flash。单价沿用仓库现价，本版至少保证 alias 生效。

#### C2. `grok-4.5` 文本价 + long context

官方分 short / long 两档。现状：`pricing.yaml` 无 long-context tier。

本版建议补最小 long-context 支持，避免 500k 模型成本系统性低估：

```yaml
grok-4.5:
  # short context
  input_per_1m: 2.00
  cached_input_per_1m: 0.30
  output_per_1m: 6.00
  long_context:
    # 以单次请求 input_tokens 是否越过阈值切换单价
    threshold_input_tokens: TBD   # 待确认官方 short/long 分界
    input_per_1m: 4.00
    cached_input_per_1m: 0.60
    output_per_1m: 12.00
```

引擎改动（小）：

- `ModelPrice` 增加可选 `LongContext`
- `computeCost`：当 `inputTokens >= threshold` 使用 long 单价
- 单测覆盖 short / long 边界

| 档位 | Input / 1M | Cached input / 1M | Output / 1M |
|---|---:|---:|---:|
| Short | $2.00 | $0.30 | $6.00 |
| Long | $4.00 | $0.60 | $12.00 |

Context window：500k（文档记载；阈值分界另确认）。

#### C3. `grok-imagine-image-quality` 按张计价

现状：

- 图片成本走 `multimodal_pricing` + token 维度
- `CostDimension` 仅支持 `tokens` / `seconds`
- 库中已有 `image_usage.image_count`、`size`、`quality`

本版补「按张」能力（与 token 通道并存）：

```yaml
multimodal_pricing:
  grok-imagine-image-quality:
    aliases:
      - grok-imagine-image
    image:
      # Media Input: Text, Image → Image
      per_image_input: 0.01
      # Output by resolution
      per_image_output:
        "1K": 0.05
        "2K": 0.07
        default: 0.05
```

| 项 | 价格 |
|---|---:|
| Media input（每张输入图） | $0.01 / img |
| Output 1K | $0.05 / img |
| Output 2K | $0.07 / img |

实现要点：

- `imageModelsCost` / `imageModelCost`：除 token 外，增加 `image_count × per_image_output[resolution]`
- resolution 从 `size` 归一：`1024*` → `1K`，`2048*` / `2K` → `2K`，未知走 `default`
- 仅 request_only / 无 `image_count` 时：`cost_known = false`
- 单测：1K / 2K / 缺 size / 缺 count

范围控制：只服务已记录 `image_count` 的 images 路径；不为了定价改写 SSE 或转发字节。

#### C4. 文档

- README / pricing 注释写清：
  - long_context 阈值假设
  - imagine 按张规则与 size 映射
  - DeepSeek alias

## 5. 架构约束（验收红线）

| 规则 | 要求 |
|---|---|
| 热路径 | proxy **零行为变更**；不因本版功能增加同步 DB / 预算检查 |
| 失败模型 | 预算 checker / 标签缺失 / 定价未知 → **只影响面板**，不影响转发 |
| 隐私 | 不接受明文 key 配置；labels 只绑 hash；不落 prompt / 响应体 |
| DB | 尽量不加表；软预算用现有 `request_usage` 聚合。若需告警去重状态，可加性新增 `key_budget_state` |
| API 兼容 | `/api/keys` 旧字段保留；新字段向后兼容 |
| 不变量 | 遵守 CLAUDE.md #1–#7；转发优先于计量完整 |

## 8. 验收用例

### Keys

- [ ] 两把不同 key 的请求分别聚合
- [ ] 成本与同 range 下 models 成本之和在「可定价模型」上一致（允许舍入）
- [ ] `key_hash` 过滤 requests 只返回该 key
- [ ] label 配置后 UI 显示昵称；未配置显示 short hash
- [ ] 不配置 labels / budgets 时行为与现网兼容

### 预算

- [ ] 低于 warn：无 budget issue
- [ ] `warn ≤ x < exceed`：`key_budget_warning`
- [ ] `x ≥ exceed`：`key_budget_exceeded`
- [ ] checker panic / DB 错不影响 `/v1/chat/completions`

### 定价

- [ ] `ds-openai-pro` 与 `deepseek-v4-pro` 同价
- [ ] `ds-openai-flash` 与 `deepseek-v4-flash` 同价
- [ ] `grok-4.5`：input 小于阈值用 short 价；≥ 阈值用 long 价
- [ ] imagine：1K / 2K 输出张数正确计价；无 count 时 partial

### 不变量

- [ ] proxy 相关测试全绿
- [ ] 无明文 key 进入 DB / 日志 / API 响应

## 12. 为什么选这一版

| 选项 | 为何不选 / 为何选 |
|---|---|
| 只做定价 | 解决不了朋友运营问题 |
| 只做 Keys 成本、无下钻 | 看见总额却无法排查 |
| 上硬拦截 | 违背近零干扰与核心不变量 |
| 上完整计费网关（用户/套餐/支付） | 过重，产品定位偏移 |
| **Keys Ops + 软告警 + 定价** | 价值 / 干扰比最高，闭环完整，数日可交付 |

## 14. 非目标重申

本版及本项目在 Key 维度上持续遵守：

- 不做 quota enforcement / 请求前额度检查
- 不做因观测失败导致的请求失败
- 不做明文 key / prompt / 响应体落库
- 不做 SaaS 多租户与支付闭环

硬控制若未来需要：放在 **外围 Gateway**，MeteringProxy 继续只提供只读账本与告警。
