# MeteringProxy v0.3.0 观测与配额设计方案

版本定位：v0.3.0 设计定稿  
日期：2026-05-05  
状态：待实施  
适用范围：MeteringProxy 观测数据模型、Responses 采集、CLIProxyAPI 旁路联动、凭证健康、配额快照、WebUI 与报表 API

本文件是 v0.3.0 的唯一有效设计入口。`docs/archive/v0.3.0-reference-only/` 中的 review 和替代设计仅作追溯参考，不作为实现约束或新增需求来源。

## 1. 核心结论

v0.3.0 不只是修复“无法识别模型”。它要正式确定 MeteringProxy 的第二条产品主线：

```text
第一主线：透明转发，尽可能不影响 LLM 体验。
第二主线：在不破坏第一主线的前提下，建立可信、可解释、可运维的观测层。
```

最终决策：

- `request_usage` 继续作为 HTTP 请求事实表，不因为 quota、side-channel 或 WebUI 查询进入转发前决策。
- 协议解析、CLIProxyAPI usage queue、凭证健康、配额快照都属于旁路观测。失败时只能降低观测完整性，不能影响 LLM 请求。
- 所有模型、用量、错误、凭证、配额信息都必须带来源语义。事实不覆盖，推断不伪装成事实。
- CLIProxyAPI 联动只做可选增强，不替代 MeteringProxy 的 HTTP 事实记录。
- 配额相关链路只走 CLIProxyAPI Management API，不接触 `/v1/*` LLM 转发链路，不做 quota enforcement、账号切换或请求前检查。
- v0.3.0 不做任何 OAuth 功能。`docs/quota-oauth-scope.md` 不能作为本版本约束来源，也不能从其中的 OAuth Scope 推导本版本工作项。
- 不保存 prompt、messages、response text、tool arguments、图片内容、文件内容、明文 API Key、明文客户端 IP、provider token、CLIProxyAPI management key、raw auth file 或 raw queue/quota payload。

## 2. 版本边界

### 2.1 必须交付

1. Responses SSE 状态机。
2. 模型来源与用量来源字段。
3. 有限请求元数据扫描，改善 `model_requested` 缺失。
4. CLIProxyAPI Management 基础客户端。
5. 基于 `/v0/management/auth-files` 的凭证健康快照，并为当前凭证/OAuth 额度状态面板提供基础数据。
6. CLIProxyAPI usage queue 可选消费。默认只落归一化事件；只有 request-id 传播被明确验证后，才允许合并补齐请求 usage。
7. 配额快照框架。完整 provider quota 查询经 `/v0/management/api-call` 或等价能力执行，并由运行时能力探测控制；产品重点是“当前各凭证额度状态”，不是深度历史。
8. WebUI/API 展示请求事实、采集状态、side-channel 状态、凭证健康与配额状态，并优化统计趋势与近期诊断体验。
9. Issues 视图加入采集缺失、usage 冲突、side-channel 断连、凭证不可用、quota 低/耗尽/刷新失败等问题。

### 2.2 不做

- 不做 quota enforcement。
- 不做请求前 quota 检查。
- 不做自动账号切换、模型 fallback 或重试调度。
- 不做 OAuth 登录、回调、状态轮询、凭证创建、凭证更新或管理中心入口。
- 不做 CLIProxyAPI 配置编辑。
- 不做通用 `/api-call` 浏览器代理。
- 不做无 request-id 的 usage 模糊匹配自动合并。
- 不保存原始 usage queue payload、quota payload、请求体或响应体。

## 3. 信息分层

v0.3.0 固定四层信息模型。

### 3.1 HTTP 事实层

来源：MeteringProxy 转发过程。

字段：

- `created_at`
- `endpoint`
- `method`
- `status`
- `latency_ms`
- `ttfb_ms`
- `stream`
- `request_bytes`
- `response_bytes`
- `api_key_hash`
- `client_ip_hash`
- `request_id`

规则：

- 这层是 MeteringProxy 最权威的数据。
- 状态码、延迟、TTFB 永远以 HTTP 事实层为准。
- side-channel 不覆盖这些字段。

### 3.2 协议解析层

来源：MeteringProxy 对请求前缀、响应 sample、SSE event 的 bounded 解析。

字段：

- `model_requested`
- `model_returned`
- `model_returned_source`
- `input_tokens`
- `output_tokens`
- `reasoning_tokens`
- `cached_tokens`
- `cache_creation_tokens`
- `total_tokens`
- `usage_source`
- `capture_outcome`
- `capture_reason`
- `terminal_event`
- `terminal_reason`
- `error_class`
- `error_type`
- `error_code`
- `error_param`

规则：

- 解析是 best-effort。
- 解析失败只能改变 `capture_outcome`、`capture_reason` 或 system counters。
- 不得为了协议解析改变上游响应字节。
- 不引入 `usage_confidence` 字段。等价语义由 `usage_source`、`capture_outcome` 和 `side_usage_events.match_status` 推导。

### 3.3 CLIProxyAPI side-channel 层

来源：CLIProxyAPI Redis-compatible usage queue。

字段：

- `provider`
- `model`
- `alias`
- `endpoint`
- `auth_type`
- `auth_index_hash`
- `source_hash`
- `api_key_hash`
- `request_id`
- token 细分
- `failed`
- `latency_ms`
- `match_status`

规则：

- 默认关闭。
- 消费 queue 与合并补齐是两个动作。可以只存事件，不合并到 `request_usage`。
- 只有 request-id 传播被明确验证时，才允许把 side event 合并到请求视图。
- 匹配不到的 side event 单独保存和展示，不自动猜测合并。
- 明文 `api_key`、`source`、`auth_index` 入口处立即 HMAC，不落库。

### 3.4 凭证健康与配额层

来源：MeteringProxy 后台 worker 通过 CLIProxyAPI Management API 只读查询。

链路独立于 LLM 请求链路：

```text
LLM traffic:
  client -> Caddy -> MeteringProxy /v1/* -> CLIProxyAPI /v1/* -> provider

Quota traffic:
  MeteringProxy background worker -> CLIProxyAPI /v0/management/* -> provider quota endpoint
```

规则：

- 凭证健康和配额快照不是请求调度依据。
- 只用于 WebUI 运维观察和问题关联。
- 查询失败不影响 LLM 转发。
- 浏览器永远拿不到 CLIProxyAPI management key、provider token、raw auth file 或 raw api-call response。

## 4. 请求相关性

CLIProxyAPI usage queue 合并的前提是 request-id 级关联。但当前本地 CLIProxyAPI 源码显示，AI API 路径的 request id 由 CLIProxyAPI 自生成，usage queue 的 `request_id` 来自其 context，不读取 `X-Request-ID` header。因此 v0.3.0 必须把“事件消费”和“请求合并”分开。

### 4.1 默认模式：passive

默认不新增请求头，只记录客户端已有的以下 header：

- `X-Request-ID`
- `Request-ID`

如果请求没有 request id，MeteringProxy 仍照常记录 HTTP 请求，但 side-channel event 不会自动合并。

### 4.2 合并门控

配置：

```yaml
observability:
  correlation:
    mode: "passive" # passive | inject_if_missing
    header: "X-Request-ID"
    side_channel_merge: "stored_only" # stored_only | request_id
    require_propagation_verified: true
```

规则：

- `stored_only`：只消费并保存 side event，不写回 `request_usage` token 字段。
- `request_id`：允许按 request id 合并，但必须满足 `require_propagation_verified=true` 且 operator 已确认 CLIProxyAPI 会把该 request id 写入 usage queue。
- `inject_if_missing` 只有在 CLIProxyAPI 支持 request-id 传播时才有意义。否则它只是改变请求透明度但换不来可用匹配能力。
- 不做 endpoint、api_key、model、时间窗口的模糊合并。

CLIProxyAPI 侧理想改动是：AI API 中间件优先读取 `X-Request-ID`，为空时再自生成。这个改动不属于 MeteringProxy v0.3.0 必须交付，但它决定 `usage_source=cliproxy_side_channel` 是否可以可靠启用。

## 5. 有限请求元数据扫描

v0.2.0 只从请求体前 4KB 提取 `model`。v0.3.0 改为两段式扫描：

```yaml
request_metadata:
  initial_bytes: 4096
  max_bytes: 65536
  extended_model_scan: false
```

规则：

- 先读取 `initial_bytes`，提取 `model`、`stream`。
- 如果 `model` 未找到，且 `extended_model_scan=true`，且 endpoint 是 LLM metered profile，则继续扫描到 `max_bytes`。
- 扩展扫描只寻找顶层 `model`，不为寻找 `stream` 继续扩大读取。
- 扫描只解析顶层 JSON key，不进入 message/content/tool arguments。
- 找到 `model` 后停止扩展扫描。
- 所有已读取字节必须通过 replay reader 原样转发。
- 如果 `model` 在 `max_bytes` 后面，记录为空。这是可接受缺失。

性能边界：

- 默认只预读 4KB，保持 v0.2.0 的转发前等待边界。
- `extended_model_scan` 默认关闭。operator 确认某些客户端经常把顶层 `model` 放在 4KB 之后时，再显式启用。
- 启用扩展扫描后最大预读 64KB。
- 慢客户端场景下扩展扫描会增加请求转发前等待，因此不得作为隐式默认优化。
- 内存上界约为并发请求数乘以 `max_bytes`，不得缓存完整请求体。

## 6. Responses SSE 状态机

当前问题的根源之一是 `/v1/responses` 只在 `response.completed.response.usage` 出现时记录模型和用量。v0.3.0 必须拆分“模型观察”和“usage 观察”。

### 6.1 状态字段

每条 Responses SSE 请求维护内存态：

```text
seen_response_created
created_model
seen_response_completed
completed_model
completed_has_usage
seen_response_failed
failed_model
seen_response_incomplete
incomplete_model
incomplete_reason
seen_error_event
usage_seen
terminal_event
terminal_reason
last_event_type
json_parse_errors
```

### 6.2 解析范围

允许解析的 SSE event：

- `response.created`
- `response.completed`
- `response.failed`
- `response.incomplete`
- `error`

允许读取的字段：

- `type`
- `response.model`
- `response.status`
- `response.usage`
- `response.incomplete_details.reason`
- `error.type`
- `error.code`
- `error.param`

不允许读取或保存：

- `response.output`
- `delta`
- `text`
- `arguments`
- `content`
- tool call payload
- error message 原文

单个 SSE event 的 JSON 解析上限默认 1 MiB。超过上限时记录 `parse_error` 或 system counter，但必须继续转发后续字节。

### 6.3 结束时映射

| 状态 | `capture_outcome` | `capture_reason` | `terminal_event` | 模型处理 |
|------|-------------------|------------------|------------------|----------|
| usage 存在 | `captured` | 空 | `response.completed` | `model_returned_source=usage` |
| completed 存在但无 usage | `skipped` | `response_completed_without_usage` | `response.completed` | 从 completed/created 记录模型 |
| incomplete 存在 | `skipped` | `response_incomplete` | `response.incomplete` | 从 incomplete/created 记录模型，`terminal_reason` 取 reason |
| created 存在但无 completed/failed/incomplete | `skipped` | `stream_ended_without_completed` | `stream_end` | 从 created 记录模型 |
| failed/error event 存在 | `failed` | `response_error_event` | `response.failed` 或 `error` | 能取 model 则记录 |
| SSE JSON 解析错误 | `failed` 或 `skipped` | `parse_error` | `stream_error` | 不影响转发 |
| 无任何可识别 event | `skipped` | `usage_not_present` | 空 | 回退到 `model_requested` |

### 6.4 flush 顺序

流式路径必须保持：

```text
read upstream chunk
write client
flush client
then parse copied bytes for observation
```

任何解析 panic、JSON 错误、行超长、sample overflow 都不能阻止下一次 read/write/flush。

## 7. 模型与用量来源

### 7.1 新增字段

```sql
ALTER TABLE request_usage ADD COLUMN model_returned_source TEXT DEFAULT '';
ALTER TABLE request_usage ADD COLUMN usage_source TEXT DEFAULT '';
ALTER TABLE request_usage ADD COLUMN terminal_event TEXT DEFAULT '';
ALTER TABLE request_usage ADD COLUMN terminal_reason TEXT DEFAULT '';
ALTER TABLE request_usage ADD COLUMN side_usage_event_id INTEGER DEFAULT 0;
```

不新增 `usage_confidence`。

### 7.2 `model_returned_source`

取值：

- `usage`
- `response_completed`
- `response_created`
- `response_failed`
- `response_incomplete`
- `http_response`
- `side_channel`
- 空

effective model 读取规则：

```text
model_returned -> model_requested -> unidentified
```

UI 文案：

| source | 展示 |
|--------|------|
| `usage` | 来自 usage |
| `response_completed` | 来自完成事件 |
| `response_created` | 来自响应事件 |
| `response_failed` | 来自失败事件 |
| `response_incomplete` | 来自中断事件 |
| `http_response` | 来自 HTTP 响应 |
| `side_channel` | 来自 CLIProxyAPI |
| request fallback | 来自请求 |
| empty | 无法识别模型 |

### 7.3 `usage_source`

DB 取值：

- `http_response`
- `cliproxy_side_channel`
- 空

API/UI 可以把空值展示为 `none`，但数据库默认仍使用空字符串兼容历史数据。

合并规则：

1. HTTP 解析到非零 usage：写入 token 字段，`usage_source=http_response`。
2. HTTP 未解析到 usage，side-channel 精确匹配且有非零 usage：写入 token 字段，`usage_source=cliproxy_side_channel`。
3. HTTP 和 side-channel 都有 usage 且数值一致：保留 HTTP token 字段，side event 标记 `matched`。
4. HTTP 和 side-channel 都有 usage 但数值不一致：不覆盖 HTTP token 字段，side event 标记 `conflict`，Issues 返回 `usage_conflict`。
5. side-channel 无 request-id 精确匹配：不合并，side event 标记 `unmatched`。

`capture_outcome` 表示 MeteringProxy 协议采集结果，不因为 side-channel enrichment 改成 `captured`。例如 HTTP SSE 没有 usage，但 side-channel 补到了 token，应显示为“HTTP 采集跳过，CLIProxyAPI 已补齐用量”。

### 7.4 `terminal_event` 覆盖范围

v0.3.0 必须完整填充 Responses API 的 `terminal_event` 和 `terminal_reason`。其他协议按低风险方式逐步填充：

- Chat Completions：如果能从流式或非流式响应安全读取 `finish_reason`，则 `terminal_event=chat.finish`，`terminal_reason` 取 `stop`、`length`、`tool_calls` 等枚举值；否则留空。
- Anthropic Messages：如果能安全读取 `message_stop` 或 `stop_reason`，则 `terminal_event=message_stop`，`terminal_reason` 取 provider 返回的 stop reason；否则留空。
- Gemini：如果能安全读取 `finishReason`，则 `terminal_event=gemini.finish`，`terminal_reason` 取 `finishReason`；否则留空。

非 Responses 协议不得为了填充 terminal 字段扩大响应解析范围。字段为空表示“本版本未可靠观测”，不是错误。

## 8. CLIProxyAPI Usage Queue 联动

### 8.1 CLIProxyAPI 前提

CLIProxyAPI 侧需要：

```yaml
usage-statistics-enabled: true
redis-usage-queue-retention-seconds: 300
```

并启用 management key。CLIProxyAPI 的 Redis-compatible queue 使用 RESP 协议，支持 `AUTH`、`LPOP`、`RPOP`。队列保留时间有限，MeteringProxy 消费慢、重启或 LPOP 后写库失败时允许丢失 side events，不做 WAL 或两阶段提交。

### 8.2 MeteringProxy 配置

```yaml
cliproxy_management:
  enabled: true
  base_url: "http://127.0.0.1:8317/v0/management"
  key_file: "/data/management_key"
  usage_queue:
    enabled: false
    resp_addr: "127.0.0.1:8317"
    queue_name: "queue"
    pop: "LPOP"
    batch_size: 50
    poll_interval: "1s"
    reconnect_interval: "5s"
    match_timeout: "10m"
    event_retention: "7d"
    merge_mode: "stored_only" # stored_only | request_id
```

`usage_queue.enabled` 默认 false，`merge_mode` 默认 `stored_only`。

如果 CLIProxyAPI 后续提供 management-side HTTP usage event 消费接口，MeteringProxy 应优先考虑 HTTP 消费模式作为 RESP 的替代或降级路径。v0.3.0 不假设该接口存在。

### 8.3 入库表

```sql
CREATE TABLE IF NOT EXISTS side_usage_events (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    received_at TEXT NOT NULL,
    received_at_unix INTEGER NOT NULL DEFAULT 0,
    request_id TEXT NOT NULL DEFAULT '',
    matched_request_usage_id INTEGER NOT NULL DEFAULT 0,
    match_status TEXT NOT NULL DEFAULT '',
    provider TEXT NOT NULL DEFAULT '',
    model TEXT NOT NULL DEFAULT '',
    alias TEXT NOT NULL DEFAULT '',
    endpoint TEXT NOT NULL DEFAULT '',
    auth_type TEXT NOT NULL DEFAULT '',
    auth_index_hash TEXT NOT NULL DEFAULT '',
    source_hash TEXT NOT NULL DEFAULT '',
    api_key_hash TEXT NOT NULL DEFAULT '',
    input_tokens INTEGER NOT NULL DEFAULT 0,
    output_tokens INTEGER NOT NULL DEFAULT 0,
    reasoning_tokens INTEGER NOT NULL DEFAULT 0,
    cached_tokens INTEGER NOT NULL DEFAULT 0,
    cache_read_tokens INTEGER NOT NULL DEFAULT 0,
    cache_creation_tokens INTEGER NOT NULL DEFAULT 0,
    total_tokens INTEGER NOT NULL DEFAULT 0,
    latency_ms INTEGER NOT NULL DEFAULT 0,
    failed INTEGER NOT NULL DEFAULT 0,
    error_class TEXT NOT NULL DEFAULT ''
);
```

`match_status` 取值：

- `stored_only`
- `matched`
- `unmatched`
- `conflict`
- `duplicate`
- `expired`
- `invalid_payload`

CPA v7.0.4 的 usage queue payload 已包含 `cache_read_tokens` 与 `cache_creation_tokens`。MeteringProxy 保存这两个 side-channel 字段；合并到 `request_usage` 时，`cache_read_tokens` 映射到既有 `cached_tokens`，`cache_creation_tokens` 映射到既有 `cache_creation_tokens`。

禁止保存字段：

- 明文 `api_key`
- 明文 `source`
- 明文 `auth_index`
- 原始 queue payload

### 8.4 匹配逻辑

匹配键：

```text
request_id
```

辅助校验：

```text
endpoint roughly matches
created_at within match_timeout
api_key_hash matches when available
```

如果 `request_id` 为空，禁止自动合并。

如果多条 side event 匹配同一 request：

- 第一条有效 usage 可合并。
- 后续 event 标记 `duplicate`，保留在 `side_usage_events`。
- 如果 CLIProxyAPI 后续支持内部 attempt id，再扩展 attempt 视图；v0.3.0 不展开多 attempt 计费。

### 8.5 保留与清理

默认保留 7 天：

```sql
DELETE FROM side_usage_events WHERE received_at_unix < cutoff;
```

清理失败只记录 system issue，不影响 LLM 请求。

## 9. 凭证健康与配额快照

本章是 v0.3.0 的自包含配额设计。实现时不要把 `docs/quota-oauth-scope.md` 当作约束来源，也不要从其中的 OAuth 讨论推导任何本版工作项。

v0.3.0 分两层交付：

- Phase A：凭证健康。只依赖 `/v0/management/auth-files`，必须交付。
- Phase B：完整配额快照。依赖 CLIProxyAPI `/v0/management/api-call` 或等价能力，以及 provider-specific adapter。运行时探测不可用时，系统降级为凭证健康，不影响其他观测功能。

当前本地 CLIProxyAPI 源码中可以看到 `POST /v0/management/api-call` 路由，但它不是所有部署版本都必然具备的公开稳定能力。因此 MeteringProxy 不能把它写死为无条件可用。

配额模块的产品优先级：

1. 首屏清楚展示当前各 provider、各凭证/OAuth auth entry 的额度状态。
2. 明确区分 `ok`、`warning`、`low`、`exhausted`、`unknown`、`stale`、`error`。
3. 能辅助诊断近期状态，例如最近刷新失败、凭证不可用、数据陈旧、provider adapter 不可用。
4. 不把深度历史作为核心功能。v0.3.0 不做 quota history 图表，数据库以当前状态为主，只保留短期刷新事件用于诊断。
5. 这里的 OAuth 只指 CLIProxyAPI 已存在的 OAuth auth file/凭证条目；MeteringProxy 不实现 OAuth 生命周期。

### 9.1 配置

```yaml
cliproxy_management:
  enabled: true
  base_url: "http://127.0.0.1:8317/v0/management"
  key_file: "/data/management_key"
  credential_health:
    enabled: true
    cache_ttl: "5m"
    timeout: "10s"
    diagnostic_retention: "72h"
  quota:
    enabled: true
    providers: ["claude", "codex", "kimi"]
    cache_ttl: "5m"
    timeout: "10s"
    concurrency: 4
    diagnostic_retention: "72h"
    low_threshold: 0.2
```

### 9.2 凭证健康查询

```text
MeteringProxy credential health poller
  -> CLIProxyAPI /v0/management/auth-files
```

只保存安全归一化后的状态：

- provider
- credential hash
- auth index hash
- label hash
- status
- success/failed 计数
- last checked time
- error class

不保存邮箱、账号名、raw label、raw auth_index、raw metadata 或 token。`label` 为空时，`label_hash` 也为空字符串，不保存空字符串的 HMAC。

### 9.3 完整配额查询

```text
MeteringProxy quota poller
  -> CLIProxyAPI /v0/management/auth-files
  -> CLIProxyAPI /v0/management/api-call
  -> provider quota/usage endpoint
```

CLIProxyAPI 在这里提供的是 management-side 便利通道：MeteringProxy 只拿 management key 调用 CLIProxyAPI，由 CLIProxyAPI 负责读取 auth files、携带 provider token 并代发固定模板的 provider quota 请求。MeteringProxy 不直接读取 raw auth file，不持久化 provider token，也不把 management key 暴露给浏览器。

约束：

- 请求模板固定在服务端。
- 浏览器不能提交任意 provider URL。
- `/api-call` 没有 URL 白名单时，MeteringProxy 必须使用内部固定模板做白名单约束。
- 每个 provider adapter 必须有 timeout。
- provider adapter 逐个验证、逐个启用。未验证或不支持的 provider 仍展示凭证健康，并在 quota 区域显示 `unknown` 或 `unsupported`，不能阻塞整个 v0.3.0。
- v0.3.0 的完整额度目标是“框架 + 已验证 provider 的当前额度”。不承诺所有 provider 一次性完美支持。
- 单个 provider 失败不能影响其他 provider。
- 失败结果写入 `quota_refresh_events`，并同步更新 `quota_current` 的 `error/stale/unknown` 状态，用于展示刷新失败。
- quota poller 只能由后台定时器、启动预热或 `/metering/api/quota/refresh` 触发，不能由 LLM 请求触发。
- `/metering/api/quota/refresh` 只排队 refresh，不同步等待 provider quota 请求完成。

## 11. WebUI 设计

### 11.0 WebUI 产品重点

v0.3.0 WebUI 的重点不是堆叠更多原始请求历史，而是：

- 统计：让 operator 快速理解当前范围内请求量、成本、token、失败、未识别模型、采集缺失。
- 趋势：把请求量、成本、token 的变化画得清楚、稳定、可比较。
- 近期诊断：把最近的错误、凭证不可用、配额不足、数据陈旧、side-channel 断连直接暴露出来。
- 当前额度：配额面板必须能一眼看到当前各凭证/OAuth auth entry 的剩余额度和可用性。

请求记录仍是诊断基础。只要当前实现没有明显性能或体验影响，v0.3.0 不为了“精简”而大幅砍掉请求明细。

### 11.1 首屏

首屏保持 v0.2.0 的克制原则，只新增一条观测健康摘要：

```text
采集正常 · side-channel 已连接（仅存储） · 凭证 1 个不可用
```

完整配额可用时：

```text
采集正常 · side-channel 已连接 · 配额 1 个 warning
```

如果 side-channel 未启用：

```text
采集正常 · side-channel 未启用 · 凭证状态正常
```

### 11.2 请求表

默认列不继续膨胀。新增详情展开或 tooltip：

- 模型来源。
- 用量来源。
- capture reason。
- terminal event。
- side usage match status。

示例：

```text
模型：gpt-5.4（来自完成事件）
用量：0（HTTP 流已完成但无 usage）
采集：跳过 / response_completed_without_usage
```

或：

```text
模型：gpt-5.4（来自请求）
用量：1,230 tokens（CLIProxyAPI 补齐）
采集：HTTP 跳过 / side-channel enriched
```

### 11.3 凭证与配额面板

新增一级模块：`Credentials` / `凭证与配额`。

设计要求参考成熟云控制台/用量控制台的模式：状态摘要在上，分组列表在下，关键风险显眼但不喧宾夺主。

默认展示：

- provider summary。
- credential table。
- stale/error 状态。
- refresh 按钮。
- last checked / next refresh。
- full quota 是否可用；不可用时说明当前只展示凭证健康。

credential table 列：

- Provider
- Credential
- Status
- Success
- Failed
- Last Checked

完整配额可用时增加：

- Plan
- Window
- Remaining
- Reset
- Quota Status

Credential 默认显示 hash 短码；不显示邮箱、账号名、label 原文、auth_index 或 token。

完整配额 UI 细节：

- 按 provider 分组，每组显示 credential count、low/exhausted/error/stale 数量。
- 每个 credential 使用一行紧凑状态条，不使用大面积装饰卡片。
- Remaining 使用进度条或细条 meter 表达，颜色只表达状态：ok、warning、low、exhausted、unknown/stale。
- `unknown` 和 `stale` 必须和 `ok` 明确区分，不能显示成绿色。
- refresh 按钮只排队刷新，点击后显示 queued/running/last checked，不让用户误以为请求已同步完成。
- 不做 quota history 图表；最多在详情里展示最近一次失败原因和最近刷新时间。
- 首次启动或后台 poller 尚未完成时显示 loading state，不显示空表冒充“没有凭证”。
- `/api-call` 不可用、provider adapter 未验证、provider 查询失败要分别显示为 `unsupported`、`unknown` 或 `error`。
- 窄屏下凭证表允许横向滚动或折叠低优先级列，但 Provider、Credential、Status、Remaining/Quota Status 必须保留。

### 11.4 趋势图表

`花费` 图表：

- 只表达成本，不绘制失败次数。
- 当前实现中 failure count 来自 timeseries bucket 的诊断字段，v0.3.0 应把失败信息移到请求健康/问题诊断区域。
- 成本 tooltip 可以显示估算状态、unpriced models、bucket 时间和 cost，不再显示 failed count，避免用户理解成失败请求也产生了花费。

`请求` 图表：

- 改为优雅的曲线趋势图，不使用生硬折线或柱状替代。
- SVG path 使用 monotone cubic、Catmull-Rom 或等价平滑算法，避免尖锐折角，同时不能产生越界或反向过冲。
- 主线表达 total requests；失败请求用细的次级线、点标记或轻量面积表达，不抢主线。
- hover/focus 不改变图表布局尺寸；tooltip 使用浮层。
- 空数据、单点数据、极低请求量时仍保持稳定视觉：单点用 marker，空数据用明确 empty state。

`Token` 图表：

- 使用完整柱状图表达每个 bucket 的 total tokens，柱子的外轮廓高度代表该 bucket 的总 token。
- 柱体内部可以用连续色带区分 input cached、input uncached、output、reasoning 等组成，但视觉上必须是一根完整柱子，不能呈现为被切成多块的碎片化卡片。
- 色带之间不使用明显 gap、粗描边或分离阴影；必要时只用细微颜色过渡或 1px 内部分隔。
- 小比例组成不强行绘制。默认低于该柱 total tokens 的 3% 或低于 2px 视觉高度的 segment 不单独画出来。
- 被隐藏的小比例组成仍计入总高度和 tooltip 明细，tooltip 必须显示完整 token breakdown。
- 如果多个小 segment 被隐藏，视觉上并入主色或剩余占比最大的 segment，避免柱子高度与 total tokens 不一致。
- 要和请求曲线、成本趋势在坐标、tooltip、空状态上保持一致。

### 11.5 请求与问题区域

当前 “请求与问题” 区域需要作为 v0.3.0 WebUI 优化项处理：

- 左右两栏顶部必须对齐，标题、摘要和内容起点保持一致。
- 两栏使用稳定 grid 布局，例如 `grid-template-columns` + 相同的 header/content 结构，避免一侧内容导致另一侧视觉下沉。
- 右侧可交互问题卡片 hover/active/focus 时不得引发任何布局位移。
- 交互态使用预留 border、outline、box-shadow 或背景色，不通过改变 border width、margin、padding、font weight 造成尺寸变化。
- 选中某个问题类型时，只更新内容区，不让整个 “请求与问题” section 高度或宽度出现抖动。
- 问题卡片有固定最小高度和稳定行高；长文本截断或换行策略必须明确。

### 11.6 Issues

Issues 按来源分组：

- Request issues。
- Capture issues。
- Side-channel issues。
- Credential issues。
- Quota issues。

默认只展示有问题的组。没有问题时显示：

```text

```

### 11.7 响应式与加载/错误状态

新增面板必须覆盖常见桌面尺寸：

- 1366x768：不出现主要信息重叠；宽表格使用横向滚动或列折叠。
- 1920x1080：主要统计、趋势、问题、配额信息在首屏或接近首屏范围内可扫读。
- 2560x1440：内容宽度受约束，不把图表和表格拉到难以阅读的超宽行。

所有依赖后台 worker 或外部 API 的模块必须有明确状态：

- `loading`：首次启动、缓存为空、刷新正在排队或运行。
- `ready`：数据新鲜且可展示。
- `stale`：缓存过期但仍可作为参考。
- `partial`：部分 provider 或部分 adapter 成功。
- `error`：当前模块不可用，但不影响其他模块。

错误态只显示安全摘要，不显示 raw provider response、token、auth metadata 或 management key。

## 12. 安全与隐私

### 12.1 允许保存

- HMAC 后的 API key。
- HMAC 后的 client IP。
- HMAC 后的 provider credential/source/auth_index。
- 模型名。
- token 数。
- provider 名。
- quota 数值。
- 安全分类后的错误类型/code/param/message。

### 12.2 禁止保存

- prompt。
- messages。
- response text。
- tool arguments。
- image URL 或 image bytes。
- file 内容。
- 明文 API key。
- 明文 provider token。
- 明文 CLIProxyAPI management key。
- raw auth file。
- raw side-channel payload。
- raw quota payload。
- raw auth metadata。

### 12.3 错误消息

沿用 v0.2.0 error message 安全策略：

- 最多 500 UTF-8 bytes。
- 去除控制字符。
- redact key-like/base64-like/long-hex。
- 只保存 provider 错误摘要或系统诊断，不保存原始 body。

### 12.4 Hash 命名空间

不同表里的 hash 不能默认互相关联：

- `request_usage.api_key_hash`：客户端调用 MeteringProxy 时使用的 API key。
- `side_usage_events.api_key_hash`：CLIProxyAPI usage queue payload 中的 API key，可能是 CLIProxyAPI/provider 侧 key。
- `credential_health.credential_hash` / `quota_current.credential_hash`：CLIProxyAPI auth-files 中的凭证稳定标识。
- `auth_index_hash`：仅用于关联 CLIProxyAPI auth entry，不展示 raw auth_index。CPA v7.0.4 的 `auth_index` 可能是字符串，进入数据库前必须 HMAC。

这些 hash 的输入来源不同，不在同一个命名空间。WebUI 不能用相同 hash 前缀暗示它们是同一个实体；后端也不能用 `api_key_hash` 做 request 与 side-channel 的自动匹配。请求与 side event 的合并只允许走已验证的 request-id 精确匹配。

```sql
ON quota_refresh_events(status, checked_at_unix);
```

### 13.2 历史数据

历史 rows：

- `model_returned_source=''`
- `usage_source=''`
- `terminal_event=''`
- `terminal_reason=''`
- `side_usage_event_id=0`

WebUI 兼容规则：

- 空 `usage_source` 且 token > 0：显示为 `http_response`。
- 空 `usage_source` 且 token = 0：显示为 `none`。
- 空 `model_returned_source` 且 `model_returned` 非空：显示为 `legacy`，UI 文案用“历史响应”。
- 空 `terminal_event`：不展示。

不回填历史 side-channel、credential health 和 quota。

## 14. 实施顺序

### Phase 1：Schema 与 API 类型

- 新增配置结构。
- 新增 DB columns/tables/indexes。
- 扩展 event/report 类型。
- 保持旧 API 字段兼容。

### Phase 2：Responses SSE 状态机

- 新增 `ResponsesStreamState`。
- 在 stream path 中先 flush 再解析。
- 提取 created/completed/failed/incomplete model。
- 记录 terminal event/reason。
- 增加 capture reason。
- 补测试和 golden fixtures。

### Phase 3：请求元数据扫描

- 用 bounded top-level JSON scanner 替换固定 4KB model extraction。
- 保持 replay transparency。

### Phase 4：CLIProxyAPI Management 与凭证健康

- 轮询 `/v0/management/auth-files`。

### Phase 5：CLIProxyAPI Usage Queue

## 15. 验收标准

### 15.1 转发透明性

- 所有新增解析测试必须验证响应字节不变。
- SSE 流 flush 顺序保持不变。
- quota、credential health 和 usage queue 全部关闭时，代理行为与 v0.2.0 一致。

### 15.2 Responses 采集

必须覆盖：

- `response.completed` 带 usage。
- `response.completed` 不带 usage，但带 model。
- 只有 `response.created`，无 completed。
- `response.failed`。
- `response.incomplete`。
- SSE event 被拆包。
- 多行 `data:`。
- 无效 JSON。
- SSE 行超长。
- SSE event JSON 超过解析上限。

预期：

- 成功有 usage：`captured`。
- completed 无 usage：`skipped / response_completed_without_usage`。
- incomplete：`skipped / response_incomplete`。
- created 后断流：`skipped / stream_ended_without_completed`。
- failed/error：`failed / response_error_event`。

### 15.3 请求元数据扫描

必须覆盖：

- 默认配置只预读 4KB，与 v0.2.0 转发前等待边界一致。
- `extended_model_scan=false` 时，4KB 后的 `model` 缺失是可接受结果。
- `extended_model_scan=true` 时，4KB 到 64KB 内的顶层 `model` 可被识别。
- 扩展扫描不进入 messages/content/tool arguments。
- 已读字节通过 replay reader 原样转发。

### 15.4 Side-channel

必须覆盖：

- stored-only 模式下只入库不合并。
- request-id 精确匹配并补齐 token。
- request-id 为空时不合并。
- request-id 匹配但 token 冲突。
- side event 重复。
- queue 断连重连。
- invalid payload。
- LPOP 后写库失败允许丢失事件。
- 明文 api_key/source/auth_index 不落库。

### 15.5 凭证健康

必须覆盖：

- auth-files 返回正常。
- auth-files 返回空列表。
- management key 无效或端点不可达。
- 单 provider 异常不影响其他 provider。
- cache stale。
- manual refresh 只排队不阻塞。
- 浏览器响应不包含 management key、provider token、raw auth file 或 raw auth metadata。

### 15.6 Quota

必须覆盖：

- `/api-call` 不可用时降级为 `phase=credential_health`。
- 配额面板主路径展示当前凭证/OAuth auth entry 状态，而不是历史趋势。
- 未验证或不支持的 provider 显示 `unsupported`/`unknown`，不阻塞其他 provider。
- `quota_current` upsert 正确表达当前状态。
- `quota_refresh_events` 只保留短期刷新诊断，不承担历史报表。
- 单 provider 成功。
- 单 credential 多窗口。
- provider 查询失败不影响其他 provider。
- cache stale。
- manual refresh 只排队不阻塞。
- 短期诊断 retention 清理。
- 固定模板约束任意 URL。
- 浏览器响应不包含 management key、provider token、raw auth file 或 raw api-call response。

### 15.7 WebUI

必须覆盖：

- 成本图只表达成本，tooltip 不显示 failed count。
- 请求图使用平滑曲线；多点、单点、空数据都有稳定表现。
- 请求图 hover/focus 不改变 SVG 或容器布局尺寸。
- Token 图每个 bucket 是完整柱子，低于阈值的小 segment 不单独绘制但仍显示在 tooltip。
- “请求与问题” 两栏顶部对齐。
- 问题卡片 hover/active/focus 无布局位移。
- 切换问题类型时 section 宽高不抖动。
- 配额状态中 `unknown`、`stale`、`error` 不能显示成正常绿色状态。
- 手动刷新配额时 UI 显示 queued/running/last checked，不同步等待 provider 查询完成。
- 配额面板覆盖 loading、stale、partial、error 状态。
- 1366x768、1920x1080、2560x1440 下主要信息不重叠，宽表格有滚动或列折叠策略。

### 15.8 性能

默认配置下：

- 默认请求元数据扫描最大 4KB；仅显式开启扩展扫描时最大 64KB。
- 非流式 response sample 继续受 `max_nonstream_sample_bytes` 限制。
- SSE 单行仍受 max line size 限制。
- 单个 SSE event JSON 解析受上限限制。
- usage queue、credential health、quota poller 不能持有代理热路径锁。
- DB 写入仍通过异步 writer 或独立低优先级后台写入，不阻塞 LLM 响应。

## 16. 决策记录

### 16.1 为什么不做模糊匹配

并发下，相同 endpoint、相同 API key、相同模型、相近时间的请求非常常见。按时间窗口自动合并会产生看起来完整但实际错误的数据。v0.3.0 只允许 request-id 精确匹配。

### 16.2 为什么 side-channel 合并要门控

当前检查到的 CLIProxyAPI 源码里，usage queue 的 `request_id` 来自 CLIProxyAPI 内部 context。MeteringProxy 注入 `X-Request-ID` 并不必然出现在 queue payload 中。把这个假设写成默认行为会产生大量看似可用但无法匹配的数据。因此 v0.3.0 默认只存 side event，合并必须由 operator 在验证后开启。

### 16.3 为什么扩展 model 扫描默认关闭

读取请求体前缀发生在转发上游之前。4KB 之后继续读到 64KB 能改善少数请求的 `model_requested` 缺失，但在慢客户端或跨地域连接下会增加用户可感知的转发前等待。v0.3.0 保留扩展扫描能力，但默认保持 v0.2.0 的 4KB 边界，由 operator 在确认收益大于延迟代价时开启。

### 16.4 为什么配额不进请求路径

配额查询慢、不稳定、provider-specific，而且可能触发额外限制。MeteringProxy 的职责是观测，不是调度。请求前 quota check 会直接破坏“尽可能不影响 LLM 体验”的核心目标。

### 16.5 为什么配额分成凭证健康和完整快照

`/v0/management/auth-files` 是明确的管理侧只读入口，能稳定提供凭证状态，足够先交付运维价值。完整配额快照需要 `/v0/management/api-call` 或等价能力，还需要 provider adapter。把两者拆开可以让 v0.3.0 不被外部版本能力卡死，同时保留完整快照作为本版配额模块的目标形态。

### 16.6 为什么不加 `usage_confidence`

`usage_confidence` 的候选值可由现有字段推导：

- `observed` = `usage_source=http_response` + `capture_outcome=captured`
- `enriched` = `usage_source=cliproxy_side_channel`
- `conflict` = `side_usage_events.match_status=conflict`
- `missing` = `usage_source=''` + token 全为 0

新增字段会增加迁移、查询和 UI 复杂度，但不提供独立信息。

### 16.7 为什么 side-channel 不覆盖 HTTP 事实

CLIProxyAPI 内部视角能解释 provider、账号和 usage，但客户端实际看到的状态码、延迟、TTFB 仍以 MeteringProxy 转发过程为准。两者是不同层级的事实，不能互相覆盖。

### 16.8 为什么不做深度 quota history

配额快照最重要的问题是“当前哪些凭证还能用、还剩多少、是否陈旧、是否刷新失败”。深度历史会增加 UI 和存储复杂度，但对日常处理 quota 耗尽、凭证失效、provider 异常的帮助有限。v0.3.0 使用 `quota_current` 表达当前状态，只保留短期 `quota_refresh_events` 用于诊断，WebUI 不做 quota history 图表，把图表预算留给请求量、成本、token 和近期问题趋势。

## 17. 最终定位

v0.3.0 完成后，MeteringProxy 的定位是：

```text
一个透明转发优先的 AI gateway observability layer。
```

它不调度、不限流、不管理凭证、不替代 CLIProxyAPI。它负责把已经发生的请求、尽力观察到的 usage、CLIProxyAPI 内部 usage、凭证健康以及 provider 配额快照放到同一套来源明确的观测模型里。
