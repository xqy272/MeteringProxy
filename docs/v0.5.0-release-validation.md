## Proxy regression comparison

The unchanged production Proxy benchmark was run on v0.4.2 and the v0.5.0 candidate on the same host, with `-benchtime=1s -count=3`. Values below are the median of the three runs.

| Scenario | v0.4.2 ns/op | v0.5.0 ns/op | Delta | v0.4.2 B/op, allocs | v0.5.0 B/op, allocs |
|---|---:|---:|---:|---:|---:|
| nonstream small | 31,152 | 32,096 | +3.0% | 65,247, 93 | 65,247, 93 |
| nonstream large | 2,358,203 | 2,350,803 | -0.3% | 1,293,126, 98 | 1,293,103, 98 |
| SSE small | 96,670 | 98,846 | +2.3% | 90,077, 95 | 90,081, 95 |
| SSE large | 34,605,291 | 34,996,279 | +1.1% | 90,956, 95 | 90,698, 95 |
| request-only large | 162,821 | 161,681 | -0.7% | 3,994, 53 | 3,993, 53 |
| concurrent, 50 workers | 20,633 | 21,028 | +1.9% | 67,657, 96 | 67,637, 96 |

Allocation counts are identical in every scenario. The small timing movement is within the observed same-host run variance; no material Proxy hot-path regression was observed. These microbenchmarks are regression evidence, not a production latency SLO.

## Privacy sentinel audit

Two requests were sent through a fresh v0.5.0 process: one successful response and one provider error. Unique synthetic sentinels were placed in:

- the plaintext client API Key;
- the prompt;
- the successful response content;
- the provider error message;
- the forwarded client IP.

The complete SQLite dump, service stdout/stderr, and Requests/Issues/Keys API responses were scanned for every sentinel. Result: zero matches.

The two stored rows had one 64-character API-key HMAC, one 64-character client-IP HMAC, and zero non-empty `error_message` values. `hash-key` produced the same stored API-key HMAC. `PRAGMA integrity_check` returned `ok`.

## WebUI live smoke

The final static assets were served by a real v0.5.0 process with `--seed-demo` and exercised through the in-app browser.

Verified:

- desktop light and dark themes;
- 390 x 844 mobile viewport, stacked Key Detail cards, internally scrollable tabs/tables, and no page-level horizontal overflow;
- Chinese and English Key list/detail content, dates, accessibility names, partial reasons, charts, and count singular/plural forms;
- language switches re-render cached Key data immediately before the async refresh, without a mixed-language interval;
- range changes and manual refresh preserve the selected Key;
- friend-a shows eight models and exactly one unpriced model; priced-but-partial models are not mislabeled as unpriced;
- partial Key cost remains `Partial` rather than a misleading `$0.00`;
- `unknown` is independently selectable, has disabled hash copy, exact scoped reports, and a clear empty Issues state;
- no browser console warning or error was captured.

The smoke found and closed two pre-release defects: mixed-language cached Key content during locale switches, and partial models being counted as unpriced in the Key Models summary.

## Migration and remaining publication gate

The separate [v0.5.0 migration and rollback rehearsal](v0.5.0-migration-rehearsal.md) proves clean upgrade, forced WAL interruption recovery, idempotent reopen, and v0.4.2 binary + pricing rollback on a synthetic million-row v0.4.2 database.

Local validation cannot prove publication state. After pushing `main` and annotated `v0.5.0`, release completion still requires:

1. all tag-triggered GitHub Actions jobs succeed;
2. GitHub Release `v0.5.0` exists and contains the Linux binary, `config.yaml`, and `pricing.yaml`;
3. GHCR publishes `v0.5.0`, `0.5.0`, `0.5`, `0`, and `latest` tags for the release digest;
4. the published image manifest and Release attachments are inspected rather than inferred from workflow configuration.
