# MeteringProxy 生产首发架构基线与实施方案

版本：v1.0-prod-baseline  
日期：2026-05-03  
状态：待执行  
用途：本文档定义 MeteringProxy 首次生产部署前必须完成的架构、数据、测试、发布与回滚基线。本文档不是无限迭代清单；功能可以有明确缺口，但底层设计必须可靠、可验证、可演进。

---

## 0. 核心结论

本轮迭代的目标不是扩展功能，而是形成生产首发基线：

- 代理透明性优先：计量逻辑不得改变上游响应语义。
- 故障隔离优先：计量失败、数据库失败、队列满不得阻塞业务请求。
- 数据可靠性优先：迁移、backfill、备份、回滚必须可重复、可解释。
- 架构边界优先：数据库结构变化不应直接泄漏到 WebUI API。
- 收敛优先：非首发必要能力进入 v1.1，不阻塞首次部署。

上线前只解决会影响生产安全、数据可维护性和后续演进的底层问题。首发后再根据真实流量和查询数据做性能优化。

## 0.1 实施批次与回滚边界

S0 是上线阻塞级别，不代表必须做成一个大提交或一次性发布。生产首发前建议按以下批次收敛，每个批次都应有独立验证和清晰回滚点：

| 批次 | 范围 | 回滚边界 |
|------|------|----------|
| A | S0.1 数据迁移、schema task、backfill 一次化 | 涉及 SQLite 文件，执行前必须备份；失败时回滚镜像，并按需要恢复数据库备份 |
| B | S0.2 WAL checkpoint、S0.3 backup/runbook 脚本修正 | 原则上不改变业务数据结构；失败时优先镜像或脚本回滚 |
| C | S0.4 Go/Docker/CI/依赖升级 | 不和数据层改动混合；失败时回滚构建链变更即可 |
| D | S0.5 全量验证与文档同步 | 不引入新功能；只确认发布门禁 |
| E | S1 可选项 | 任一项范围扩大时拆到 v1.1 |

涉及数据库的批次必须记录三件事：上线前镜像 tag、上线前 SQLite 备份文件、是否需要数据库恢复才能回滚。工具链升级必须作为独立批次处理，避免和迁移行为混在一起后无法快速定位回归来源。

## 2. 生产首发不变量

所有实现都必须遵守以下不变量：

1. 非计量路径必须保持透明转发。
2. 计量路径不得因为解析失败、写库失败、队列满而影响响应转发。
3. 数据库迁移必须幂等；同一版本重复启动不能造成重复副作用。
4. backfill 必须可追踪；已完成任务不得在每次启动重复全表扫描。
5. 健康指标是可观测性数据，不得成为主请求路径依赖。
6. 回滚时必须能恢复到上一镜像，并能判断是否需要恢复 SQLite 文件。
7. salt 文件是历史 hash 分组能力的根，必须持久保存、加密备份、不可编辑、不可随意轮换。
8. 生产环境同一个 SQLite 文件只能由一个 MeteringProxy 实例写入；部署和回滚必须先停止旧实例，再启动新实例。
9. 默认不记录请求体、响应体、prompt、明文凭证或完整 API key；排障日志必须脱敏、限时、可关闭。
10. 首发不引入未被测试覆盖的新抽象。

## 3. S0 实施方案

### S0.1 数据迁移与 backfill 一次化

#### 当前问题

当前 `internal/db/db.go` 使用 `ensureColumns + createIndexes + runBackfills`：

- `ensureColumns` 对加列场景是合理的。
- `createIndexes` 是幂等的。
- `runBackfills` 每次启动都会执行 30+ 条 `UPDATE`，在大表上会反复扫描。
- `schema_migrations` 当前只记录 `schemaVersion`，还没有承担迁移链和任务状态职责。

#### 设计目标

首发不需要一次性重写全部迁移历史，但必须建立正确的演进框架：

- 结构迁移由 `schema_migrations` 记录。
- 数据修复任务由 `schema_tasks` 记录。
- `ensureColumns` 保留为兼容安全网。
- backfill 按版本拆分成一次性 task，而不是每次启动固定执行。

#### 表设计

`schema_migrations` 用于结构版本：

```sql
CREATE TABLE IF NOT EXISTS schema_migrations (
    version INTEGER PRIMARY KEY,
    name TEXT NOT NULL DEFAULT '',
    applied_at TEXT NOT NULL
);
```

`schema_tasks` 用于一次性数据任务：

```sql
CREATE TABLE IF NOT EXISTS schema_tasks (
    name TEXT PRIMARY KEY,
    applied_at TEXT NOT NULL
);
```

如果旧库已有 `schema_migrations(version, applied_at)`，通过 `ensureColumns` 或专用 helper 补齐 `name TEXT NOT NULL DEFAULT ''`。

#### 推荐 migrate 流程

`migrate(sqlDB)` 的顺序应固定为：

1. 设置 SQLite PRAGMA。
2. 创建 `schema_migrations` 和 `schema_tasks`。
3. 创建基础业务表，使用 `CREATE TABLE IF NOT EXISTS`。
4. 执行未来非幂等结构迁移。
5. 执行 `ensureColumns`，补齐历史库缺失列。
6. 执行 `createIndexes`。
7. 执行一次性 backfill task。
8. 写入当前 `schemaVersion` 和 `PRAGMA user_version`。

推荐 helper：

```go
func runSchemaTask(sqlDB *sql.DB, name string, fn func(*sql.Tx) error) error {
    tx, err := sqlDB.Begin()
    if err != nil {
        return err
    }
    defer tx.Rollback()

    var existing string
    err = tx.QueryRow(
        "SELECT name FROM schema_tasks WHERE name = ?",
        name,
    ).Scan(&existing)
    if err == nil {
        return nil
    }
    if err != sql.ErrNoRows {
        return err
    }
    if err := fn(tx); err != nil {
        return err
    }
    if _, err := tx.Exec(
        "INSERT INTO schema_tasks (name, applied_at) VALUES (?, ?)",
        name,
        time.Now().UTC().Format(time.RFC3339),
    ); err != nil {
        return err
    }
    return tx.Commit()
}
```

backfill 必须按版本拆分。当前 `runBackfills` 应重命名为 `backfillV4Normalization`，含义是“把任意 legacy DB 的历史数据标准化到 v4 基线”。该函数接收 `*sql.Tx`，所有 SQL 在同一事务内执行。未来新增 v5 backfill 时，必须新增独立函数和独立 task name，不能继续把新逻辑塞进已标记完成的 v4 函数。

```go
if err := runSchemaTask(sqlDB, "backfill_v4_normalization", backfillV4Normalization); err != nil {
    return err
}

// Future example:
if err := runSchemaTask(sqlDB, "backfill_v5_xxx", backfillV5Xxx); err != nil {
    return err
}
```

注意：`schema_tasks` 适合当前这类幂等 backfill。即使任务本身幂等，也应把任务执行和完成标记写入放在同一事务中；否则可能出现 backfill 成功但 task 标记失败，导致下次启动再次全表扫描。未来如果任务过大、需要分片或引入非幂等数据迁移，必须设计带进度的可恢复迁移步骤，而不是只依赖一个完成标记。

#### 迁移链边界

首发可以先保留当前 `ensureColumns` 体系，不强行把历史版本拆成完整 v1/v2/v3/v4 迁移。真正需要非幂等结构变更时，再新增：

```go
type migration struct {
    version int
    name    string
    up      func(*sql.DB) error
}
```

每个 migration 必须满足：

- 在事务中执行，SQLite 不支持事务的操作除外。
- 成功后写入 `schema_migrations`。
- 失败时返回错误，阻止服务启动。
- 有 legacy DB 测试覆盖。

#### 验收标准

- 新库首次启动会执行 backfill task，并写入 `schema_tasks`。
- 同一个库第二次启动不会再次执行 backfill SQL。
- 每个 backfill task 对应一个版本化函数和一个唯一 task name。
- 旧版 SQLite 文件仍能打开、补列、查询、写入。
- `PRAGMA user_version` 等于当前 `schemaVersion`。

### S0.2 WAL 维护策略

#### 当前问题

项目使用 SQLite WAL 模式。WAL 不一定“无限增长”，SQLite 也有自动 checkpoint；但长期运行、长读事务、备份、异常读连接等场景下 WAL 文件仍可能增长，需要显式维护和观测。

#### 设计目标

- checkpoint 不能影响代理主路径。
- 默认使用低侵入的 `PASSIVE`。
- 只有 WAL 文件超过阈值时尝试 `TRUNCATE`。
- checkpoint 失败只记录日志，不阻止服务运行。

#### DB 方法

推荐新增：

```go
type WALCheckpointResult struct {
    Busy        int
    LogFrames   int
    Checkpointed int
}

func (db *DB) CheckpointWAL(mode string) (WALCheckpointResult, error) {
    pragma := "PRAGMA wal_checkpoint(PASSIVE)"
    switch mode {
    case "", "PASSIVE":
        pragma = "PRAGMA wal_checkpoint(PASSIVE)"
    case "TRUNCATE":
        pragma = "PRAGMA wal_checkpoint(TRUNCATE)"
    default:
        return WALCheckpointResult{}, fmt.Errorf("unsupported wal checkpoint mode: %s", mode)
    }
    var r WALCheckpointResult
    err := db.sql.QueryRow(pragma).Scan(&r.Busy, &r.LogFrames, &r.Checkpointed)
    return r, err
}
```

`mode` 使用白名单，不接受任意外部输入，避免拼接 SQL 风险。

#### 调度策略

健康指标仍每 60 秒写入一次；WAL checkpoint 不必每 60 秒执行。

推荐策略：

- 每小时执行一次 `PASSIVE` checkpoint。
- 如果 `usage.sqlite-wal` 超过 `128 MiB`，尝试一次 `TRUNCATE`。
- 如果返回 busy 或出错，只 log，不重试风暴。
- WAL 文件大小可以在 `main.go` 中用 `os.Stat(cfg.Database + "-wal")` 判断；如果希望封装到 `db.DB`，则需要在 `DB` 结构中保存数据库路径。

示意：

```go
lastCheckpoint := time.Now()
case <-ticker.C:
    if time.Since(lastCheckpoint) >= time.Hour {
        result, err := database.CheckpointWAL("PASSIVE")
        if err != nil {
            log.Printf("wal checkpoint error: %v", err)
        } else {
            log.Printf("wal checkpoint: busy=%d log=%d checkpointed=%d",
                result.Busy, result.LogFrames, result.Checkpointed)
        }
        lastCheckpoint = time.Now()
    }
    // 继续写 health metric
```

#### 验收标准

- checkpoint 逻辑不在请求路径中。
- checkpoint 错误不会导致服务退出。
- 日志能看到 checkpoint 结果。
- 大 WAL 文件场景有可执行的维护策略。

### S0.3 部署、备份与回滚 runbook

#### 部署模式与单实例约束

生产首发使用单实例 SQLite 写入模型。同一个 `usage.sqlite` 文件不得被两个 MeteringProxy 实例同时写入。

部署和回滚必须采用 stop-then-start：

```bash
docker stop metering-proxy
docker rm metering-proxy
```

不采用滚动部署同一挂载目录。如果未来需要零停机发布，应先引入外部数据库或明确的实例级文件锁。SQLite 在本地文件系统上有锁机制，但本项目不把“多个进程共享同一 DB 写入”作为受支持部署形态；该形态会带来写锁竞争、重复 health metric、回滚状态不清和网络文件系统风险。

#### Docker 挂载目录权限

Docker 镜像内进程以 UID/GID `1000:1000` 运行。使用宿主机目录挂载时，必须确保容器用户可读写数据目录：

```bash
mkdir -p /opt/ai-gateway/metering
chown -R 1000:1000 /opt/ai-gateway/metering
chmod 700 /opt/ai-gateway/metering
```

裸机 systemd 部署可以使用宿主机服务用户（例如 `www-data`），但不要把 Docker 的 UID 1000 权限假设和 systemd 用户混用。同一部署方式内保持数据目录 owner 一致。

#### salt 文件策略

salt 文件优先级至少等同 SQLite 数据库文件，且更难恢复：

- salt 丢失后，历史 API key/IP hash 无法继续关联。
- salt 泄漏后，低熵输入（例如部分 IP）存在被字典反推的风险。
- salt 不应轮换、编辑或由部署脚本覆盖。
- salt 必须随数据库一起加密备份。

首次部署前必须确认 salt 文件存在、权限受限且已纳入备份。

#### 上线前备份

首次部署前必须确认：

- SQLite 数据目录存在且权限受限。
- salt 文件存在且已备份。
- `config.yaml` 与 `pricing.yaml` 已冻结。
- 当前镜像 tag、配置文件、数据库文件路径已记录。
- `pricing.yaml` 只在进程启动时加载；修改定价后必须重启服务才会刷新成本估算。

备份命令：

```bash
DB_PATH=/opt/ai-gateway/metering/usage.sqlite \
BACKUP_DIR=/opt/ai-gateway/metering/backups \
RETENTION_DAYS=90 \
MONTHLY_KEEP=12 \
/path/to/scripts/backup.sh
```

如果在 Docker 容器内手动备份：

```bash
backup_file="./backup-$(date +%Y%m%d-%H%M%S).sqlite"
docker exec metering-proxy sqlite3 /data/usage.sqlite ".backup /tmp/backup.sqlite"
docker exec metering-proxy sqlite3 /tmp/backup.sqlite "PRAGMA integrity_check"
docker cp metering-proxy:/tmp/backup.sqlite "$backup_file"
gzip "$backup_file"
```

#### 备份完整性验证

`scripts/backup.sh` 必须在压缩和移动备份文件前执行完整性检查，并确认输出为 `ok`：

```bash
integrity="$(sqlite3 "$TMP_DB" "PRAGMA integrity_check")"
if [[ "$integrity" != "ok" ]]; then
    echo "Backup integrity check failed: $integrity" >&2
    exit 1
fi
```

同时建议将备份命令改为不依赖单引号拼接路径：

```bash
sqlite3 "$DB_PATH" ".backup \"$TMP_DB\""
```

如果完整性检查失败，本次备份必须失败退出，不得生成最终 `.gz` 文件。

#### 上线后验证

部署完成后必须验证：

```bash
curl -s http://127.0.0.1:8320/metering/api/health
curl -s http://127.0.0.1:8320/metrics
ls -lh /opt/ai-gateway/metering/usage.sqlite*
```

观察项：

- `queue_depth` 应能回落。
- `dropped_events_total` 正常情况下应为 0。
- `db_write_errors_total` 应为 0。
- `parse_errors_total` 应低且可解释。
- WAL 文件不应持续无界增长。
- 业务请求必须继续正常转发。
- 修改 `pricing.yaml` 后必须重启进程；未重启时 WebUI cost 仍使用启动时加载的价格表。

#### 回滚策略

回滚分两类：

1. 仅镜像回滚：代码异常但数据库迁移兼容，直接恢复上一镜像。
2. 镜像 + 数据库恢复：迁移导致数据不可用，先停服务，再恢复上线前备份。

镜像回滚：

```bash
docker stop metering-proxy
docker rm metering-proxy
docker run -d \
  --name metering-proxy \
  --restart unless-stopped \
  -v /opt/ai-gateway/metering:/data \
  --add-host host.docker.internal:host-gateway \
  -p 127.0.0.1:8320:8320 \
  ghcr.io/xqy272/ai-gateway-metering-proxy:<previous-tag> \
  -config /data/config.yaml
```

数据库恢复必须先停止服务：

```bash
docker stop metering-proxy
rm -f /opt/ai-gateway/metering/usage.sqlite-wal /opt/ai-gateway/metering/usage.sqlite-shm
gunzip -c backup-YYYYMMDD-HHMMSS.sqlite.gz > /opt/ai-gateway/metering/usage.sqlite
chown 1000:1000 /opt/ai-gateway/metering/usage.sqlite
chmod 600 /opt/ai-gateway/metering/usage.sqlite
```

WAL 模式下不能只覆盖主库文件却保留旧的 `usage.sqlite-wal` 和 `usage.sqlite-shm`。恢复备份前必须先停止服务并删除旧 WAL/SHM 文件，让 SQLite 基于恢复后的主库重新创建。上面的 `chown 1000:1000` 适用于 Docker 部署；如果使用 systemd 裸机部署，应改为对应服务用户。

恢复后再启动旧镜像并验证 `/metering/api/health`。

### S0.4 构建链升级与一致性

#### 当前状态

当前代码基线：

- `go.mod`：Go 1.21
- `Dockerfile`：`golang:1.21-alpine` + `alpine:3.20`
- GitHub Actions：Go 1.21
- `github.com/mattn/go-sqlite3`：v1.14.22

截至 2026-05-03，Go 官方稳定分支已是 1.26.x 与 1.25.x；Alpine 3.20 已过官方支持日期。首发不应继续使用过期基线。

#### 推荐目标

生产首发采用保守升级：

| 组件 | 目标 | 原因 |
|------|------|------|
| Go | 1.25.x | 当前仍受支持，相比 1.26.x 更保守 |
| go-sqlite3 | v1.14.44 | 当前可用最新 patch，降低已修复问题风险 |
| Docker build | `golang:1.25-alpine` | 与 Go 目标一致 |
| Docker runtime | `alpine:3.23` | 当前受支持的 Alpine 分支 |
| GitHub Actions | `go-version: "1.25.x"` | CI 与构建一致 |

如果选择 Go 1.26.x，也必须作为独立升级项执行完整回归，不和数据层改动混在一起。

#### 必改文件

- `go.mod`
- `go.sum`
- `Dockerfile`
- `.github/workflows/build.yml`

执行：

```bash
go get github.com/mattn/go-sqlite3@v1.14.44
go mod edit -go=1.25
go mod tidy
go test ./...
```

## 4. S1 实施方案

### S1.1 WebUI/API 使用领域 report 类型

#### 当前问题

`internal/webui/server.go` 依赖 `store.ReportStore`，但 handler 仍直接返回 `db.*Row`。这会让数据库 row 类型成为 API 响应契约的一部分。

#### 设计目标

首发目标是稳定 API 响应类型，而不是一次性重写 read-side store。

推荐做法：

- WebUI handler 返回 `event.*Report`。
- `event/mapping.go` 补齐 `RequestsFromDB`、`ErrorTimelineFromDB`、`HealthFromDB`。
- `ModelsFromDB` 只做 DB row 到 report 的转换，不直接依赖 `pricing.Pricing`。
- 成本计算仍放在 WebUI handler 或后续 report service 中。

原因：`event` 包不应反向依赖 pricing，避免领域类型层承担展示计算职责。

设计注记：首发可以先沿用 `event.*Report` 来控制改动范围，但不要让 `event` 包长期同时承担写入事件和 read-side report 的全部职责。如果 v1.1 继续扩展报表、过滤、导出或复杂展示计算，优先新建 `internal/report` 或 `internal/readmodel` 包承载查询模型，避免 `event` 包膨胀成混合层。

#### 验收标准

- `/api/summary`、`/api/models`、`/api/keys`、`/api/timeseries`、`/api/requests`、`/api/errors` 返回结构不变。
- WebUI 测试通过。
- `event/mapping.go` 中没有未使用或误导性的转换函数。

### S1.2 store/main 边界清理

#### 当前问题

`main.go` 直接调用 `database.InsertHealthMetric(...)`。这不影响功能，但绕开了 `store.HealthWriter` 接口。

#### 推荐做法

保持 `db.DB` 为具体对象，用接口约束调用点：

```go
sqliteDB, err := db.Open(cfg.Database)
if err != nil {
    log.Fatalf("Failed to open database: %v", err)
}
defer sqliteDB.Close()

var reportStore store.ReportStore = sqliteDB
var healthWriter store.HealthWriter = sqliteDB
eventSink := store.NewEventSink(sqliteDB)
```

使用：

- `eventSink` 给 writer。
- `reportStore` 给 WebUI。
- `healthWriter` 给 health reporter。

不要为了抽象而隐藏 `Close()`；生命周期管理仍由 `main.go` 持有具体 `sqliteDB`。

### S1.4 代理防御性修补

#### status 0 归一化

透明转发路径已有 `status == 0` 防御。计量路径也应在 `RoundTrip` 后统一归一化：

```go
status := resp.StatusCode
if status == 0 {
    status = http.StatusOK
}
```

后续 `handleStream`、`handleNonStream`、`recordUsage` 使用该 `status`，避免记录异常状态码。

#### SSE skip 可观测性

当前 `sse_line_skips_total` 是 Prometheus counter。是否持久化到 `health_metrics` 属于 S1，不阻塞上线。

如果持久化：

- 只保存累计计数，不保存原始 SSE 行内容。
- 原始内容可能包含响应数据，不应进入数据库。
- `health_metrics` 增加 `sse_line_skips_total INTEGER DEFAULT 0`。
- `metrics` 增加 `SSELineSkips()` getter。
- `InsertHealthMetric` 和 `store.HealthWriter` 签名同步更新。

## 5. 不建议首发修改的设计

### 复合索引

首发不盲目新增复合索引。原因：

- 写入路径会承担额外索引维护成本。
- 当前查询主要是时间范围聚合，是否命中复合索引需要真实数据验证。
- 请求列表过滤的索引顺序需要依据查询模式决定，例如 `model_returned = ? AND created_at_unix >= ?` 更可能受益于 `(model_returned, created_at_unix)`，而不是固定 `(created_at_unix, model_returned)`。

首发后用真实数据库执行：

```sql
EXPLAIN QUERY PLAN
SELECT ...
```

再决定是否添加定向索引。

### Billable 字段

`billable_*` 字段继续保留，不在首发启用 event-time pricing。

启用前必须先决定：

- 历史数据一次性 backfill。
- 查询时 `COALESCE(billable_total, query_time_cost)` 渐进过渡。
- pricing.yaml 是否引入 endpoint 维度。

### profile 配置化

profile registry 当前内置路径清晰、测试面小。配置化会引入配置校验、错误恢复和兼容性问题，不适合首发前加入。
